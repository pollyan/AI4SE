# 设计原则与核心信念 (Design Principles)

本指南列出了 AI4SE 架构中的核心哲学和顶层设计原则，引导开发过程中的工程决策，无论是对人类工程师还是对 AI 智能体（Agents）。

---

## 核心信念 (Core Beliefs)

### 1. Agent-First 操作原则
**“Agent 看不到的东西，对它来说就不存在”**。
为了让 AI 智能体能够独立解决复杂任务并充分理解架构意图，所有知识必须在代码库内显式表达：
- 不要依赖隐藏在聊天/Slack中的讨论上下文。
- 把高层次想法转化为目录里的 Markdown 文档或可执行代码。

### 2. 仓库即知识体系 (The Repository is the System of Record)
代码库内的结构化目录 (`docs/`) 应当作为真实情况的唯一记录。文档不再是没有生命的附加物，而是 Agent 读取、学习和规划的原料。
- 文档过期会导致所谓的"吸引性公害"（attractive nuisance），比没有文档更糟糕。
- 必须保持文档与代码同步。

### 3. 机械化优于文档约束
人类可能因为文档的警告而谨慎行事，但要大规模扩展 Agent 的能力，必须把规则和边界变为**机械强制（Mechanically Enforced）**。
- 如果我们限制项目依赖的层级方向，请使用 linter 或架构测试代码来阻拦，而不是仅仅在文档里说明。 

---

## 项目与架构特性要求

### 个人 Demo 项目：确定性与简洁优先
本项目是个人学习和演示项目，不包含需要保证大规模向后兼容或历史包袱的生产级复杂性：
- **确定性优先**：严格要求结构与契约，若有错误则直接抛出拦截，**拒绝静默修补或降级**。
- **及早暴露问题**：如果有问题直接报错，不掩盖异常。
- **不保留废弃代码**：废弃的 API 端点或旧版协议应**直接删除**，不要设置兼容层。
- **简洁优先**：避免过度设计。不必要的抽象层和适配层只会增加系统的不可见混乱。

### Artifact 格式分离原则
为了确保 AI 产生的复杂 Artifact (如测试用例、PRD 等) 便于系统处理或 UI 解析，**格式的约束应由数据模型驱动，而不硬编码在 Prompt 中。**

- **数据模型层 (Data Model)**: 在 `models.py` 内部规定结构的字段、枚举、类型定义。
- **渲染逻辑层 (Rendering Logic)**: 在工具函数或视图层组装为特定输出（如 Markdown）。
- **Prompt (提示词) 职责**: 仅声明逻辑目标（“做什么”与“判断标准”），绝对不写入冗长甚至易出错的字符级排版示例与格式细节。
