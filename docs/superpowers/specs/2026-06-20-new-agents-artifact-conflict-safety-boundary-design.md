# New Agents Artifact 冲突安全边界收口设计

## 用户故事

作为正在人工校准右侧产出物的用户，当我基于旧版本保存草稿、同时服务端已有新版本时，我希望系统只在能够证明安全的场景下提供自动合并；如果无法证明安全，系统必须明确告诉我为什么需要手工处理，并确认我的草稿仍被保留。

## 背景

当前 Artifact 冲突处理已经覆盖大量安全自动合并场景，包括章节改写、章节移动、章节增删、章节重命名、段落移动、结构化块重排、同章节段落改写/删除/插入等。用户主路径已经不是“再补一个 merge 算法”，而是需要一个可信边界：不能安全自动合并时，不应只让自动合并按钮消失，而应给出可理解的人工处理路径。

## 范围

- 在 `ArtifactPane` 保存冲突态中，为无法证明安全的复杂冲突显示统一人工处理原因。
- 保留已有更具体的拒绝原因：
  - `结构化块重排需人工处理`
  - 同章节多处段落插入/改写的 `自动合并暂不可用`
- 保留已有安全自动合并入口，不让通用拒绝原因覆盖 `自动合并非重叠变更`。
- 明确草稿保留，并引导用户打开 `对比服务端版本` 后手工确认。
- 更新 `docs/todos/new-agents-ux-professionalization.md`，把更复杂三方 merge 的后续口径收敛为“只在完整、可证明安全的用户冲突场景下继续做”。

## 非目标

- 不新增新的三方 merge 算法分支。
- 不改服务端 artifact update API。
- 不做多人实时协同、分享权限、恢复中心或 intent-tester 打通。
- 不重启高保真 PDF 图片级导出。

## 关键场景

### 场景 1：复杂重叠冲突转人工

用户编辑草稿时，服务端和草稿都改写了同一章节内多个段落，系统无法证明顺序和语义安全。冲突卡片应显示 `自动合并暂不可用`，说明双方改动存在重叠或顺序无法证明安全，草稿已经保留，需要打开 `对比服务端版本` 手工确认。

### 场景 2：安全自动合并不受影响

如果冲突属于现有可证明安全的非重叠合并场景，仍显示 `自动合并非重叠变更`，不显示通用拒绝原因。

### 场景 3：具体拒绝原因优先

如果冲突属于结构化块重排，继续显示 `结构化块重排需人工处理`，因为它比通用拒绝原因更能解释风险。

## 验收条件

1. 复杂不可自动合并冲突中，用户看到 `自动合并暂不可用` 和人工处理说明。
2. 同一冲突中，`自动合并非重叠变更` 不出现，`对比服务端版本` 仍出现。
3. 用户编辑草稿 textarea 的内容保持为保存前草稿。
4. 已有安全自动合并测试继续通过。
5. 结构化块重排拒绝原因继续优先显示。

## 风险

- `ArtifactPane.tsx` 文件较大，通用拒绝原因如果条件过宽，可能覆盖安全自动合并入口。实现必须以 `!autoMergedConflict` 为前置。
- 通用拒绝原因如果在非冲突态显示，会干扰正常编辑。实现必须以 `conflictArtifact` 为前置。
- 过度新增算法会造成长尾膨胀。本轮只做安全边界，不新增自动合并能力。
