# New Agents Artifact 跨章节段落移动自动合并设计

## 背景

Artifact 冲突合并已经支持非重叠补充、章节改写、章节移动、章节增删、章节重命名和同章节普通段落移动。当前跨章节段落移动会被 `hasCrossSectionUnitMovement` 识别并阻止章节改写 fallback，但没有后续合并器，因此用户把一个普通段落移动到更合适章节时，仍需要手工处理冲突。

## 用户故事

作为正在校准右侧产物的用户，当我只是把一个普通段落从一个章节移动到另一个章节，而模型或服务端同时更新了其他非冲突章节时，我希望系统能自动合并，保留我的信息架构调整和服务端新增判断，减少逐行处理成本。

## 范围

- 支持唯一普通段落块跨章节移动。
- 支持一方移动段落，另一方只改写非源/目标章节。
- 支持双方把同一个普通段落移动到同一个目标章节位置。
- 合并后继续写入 `artifact_auto_merge_applied` 活动轨迹，summary 使用 `非重叠跨章节段落移动`。
- 保持现有冲突 UI 和保存 API 不变。

## 非目标

- 不支持列表项、表格行、fenced code block、Mermaid block 或结构化可视化块内部重排。
- 不支持移动段落同时被改写。
- 不支持双方移动不同段落或移动到不同目标位置。
- 不支持跨章节移动同时改写源章节或目标章节的其他段落。
- 不修改 backend、store、SSE、Markdown renderer、PDF/DOCX 导出。

## 合并规则

1. base、server、draft 的 preamble 必须一致，Markdown 标题结构必须一致。
2. 被移动的段落块必须在 base 全局唯一，且不是 unsafe block。
3. 移动侧只能表现为一个段落块从源章节转移到目标章节；其他段落块保持原章节归属。
4. 非移动侧如果改写源章节或目标章节，则不自动合并。
5. 非移动侧可以改写其他章节；这些改写会与移动侧的跨章节移动一起保留。
6. 双方移动同一个段落到同一个章节位置时可以合并；移动目标不同则不合并。

## 验收条件

1. draft 跨章节移动一个普通段落、server 改写其他章节时，出现 `自动合并非重叠变更`，点击后草稿同时保留移动和 server 改写。
2. server 跨章节移动一个普通段落、draft 改写其他章节时，同样可以自动合并。
3. 双方移动同一段落到同一目标位置时可以自动合并。
4. 移动段落被改写、段落重复、列表/表格/fenced block 跨章节移动、双方移动不同段落时不出现自动合并入口。
5. 既有同章节段落移动、章节移动、章节重命名、章节增删测试继续通过。
