排序结论：
1. 选择 A，因为它直接延续已经完成的 Artifact 自动合并能力链，能用一个用户可见动作消化“章节集合变化”的相邻缺口，且可用现有组件测试稳定验收。
2. B 暂不选，因为重命名和新增+删除在 Markdown 结构上相似，保守规则不足时容易误合并。
3. C 暂不选，因为导出高保真属于另一个用户动作链，当前切片先收敛协作冲突。

切片厚度门禁：
- 入口：现有保存冲突卡片中的 `自动合并非重叠变更`。
- 动作：用户点击自动合并。
- 系统处理：三方 Markdown 章节解析，保守合并新增/删除章节和非冲突章节改写。
- 可见结果：编辑草稿更新为合并后的 Markdown。
- 状态承接：继续使用现有保存流程和 `artifact_auto_merge_applied` 活动轨迹。
- 失败反馈：歧义场景不显示自动合并入口，保留现有对比服务端版本和手工逐行/块级处理。
- 证据：新增 RED/GREEN 测试、完整 `ArtifactPane` 测试、lint/build/diff check。

## 用户故事

作为 Lisa / Alex 的工作流用户，当我在右侧产出物中新增一个完整章节，或删除一个已经不需要且服务端没有同时改动的章节时，如果保存时刚好服务端更新了其他章节，我希望系统能自动合并这些非重叠变更，而不是要求我逐行采纳或重新复制内容。

## 范围

进入本轮：
- 支持唯一 Markdown 标题章节的保守新增合并：例如服务端改写 `## 风险策略`，用户新增 `## 覆盖策略`。
- 支持唯一 Markdown 标题章节的保守删除合并：例如用户删除 `## 交付计划`，服务端只改写 `## 风险策略`，且被删除章节未被服务端改写。
- 继续保留服务端新增章节和用户新增章节，顺序以服务端章节顺序为基准，并在合适位置追加用户新增章节。
- 自动合并后记录 `artifact_auto_merge_applied`，summary 为 `合并轨迹：自动合并服务端与草稿的非重叠章节增删`。

不进入本轮：
- 不自动合并章节重命名。
- 不自动合并重复标题。
- 不自动合并双方都改写同一章节。
- 不自动合并服务端改写了用户删除的章节。
- 不处理段落级移动、列表项移动、跨标题层级重组或语义相似度判断。
- 不修改后端 artifact update API、run snapshot、workflow runtime 或 prompt。

## 验收条件

1. 当服务端改写既有章节、用户新增一个章节时，冲突卡片显示 `自动合并非重叠变更`。
2. 点击自动合并后，编辑草稿包含服务端改写、用户新增章节和未变章节，并记录 `非重叠章节增删` 活动轨迹。
3. 当用户删除一个未被服务端改写的章节、服务端改写其他章节时，点击自动合并后草稿保留服务端改写并应用用户删除。
4. 当章节集合变化表现为重命名歧义，或服务端改写了用户删除的章节时，不显示自动合并入口。
5. 既有插入、章节改写、章节移动自动合并测试继续通过。
