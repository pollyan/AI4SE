# 目标模式待办收口与归档生命周期设计

- 日期：2026-07-10
- 状态：设计已获用户方向确认，待书面复核
- 目标文件：`docs/strategy/goal-mode-playbook.md`
- 目标：当一份长期 todo 的全部厚切片进入终态后，目标模式必须在进入 `NEXT` 前完成归档和活跃入口清理，避免完成待办继续被扫描为当前候选。

## 1. 背景与问题

当前 Playbook 已经区分活跃 todo 与历史记录，也要求目标模式更新稳定事实源，但没有定义 todo 从“活跃”进入“归档”的完整生命周期。因此会出现以下失真：

1. 实现已经在另一分支或远端完成，当前分支仍保留执行前草稿。
2. todo 的全部切片已经完成，但文件仍位于活跃目录、索引仍把它列为当前入口。
3. 后续 `BOOTSTRAP` 只看到未勾选 checkbox，就把已完成能力重新加入 CGA 候选。
4. 单个文件的“已完成”状态、当前 `HEAD`、远端提交和活跃索引可能互相矛盾。

本设计把归档定义为 `DELIVER` 的完成门禁，而不是可选的文档整理。归档必须以当前事实版本为依据，并和活跃入口清理一起完成。

## 2. 目标与非目标

### 2.1 目标

- 规定多切片 todo 的归档粒度为“整份 todo 收口后归档”。
- 明确切片终态、todo 终态和允许归档的事实条件。
- 要求归档候选中的完成实现已经位于当前分支的已验证待交付工作树，不能只存在于未合流分支；离开 `DELIVER` 时，新 `HEAD` 必须同时包含实现事实与归档事务。
- 要求归档、索引清理、未来引用迁移和完成证据形成一个原子交付边界。
- 保留历史引用，不为了消除旧路径而改写过去执行事实。
- 阻止 `BOOTSTRAP`、CGA 或目标承接检查从归档记录恢复实现。
- 将规则融合进当前十状态 Playbook，不新增状态、不复制旧模板。

### 2.2 非目标

- 不在本轮归档任何现有 todo。
- 不在本轮解决本地 `master` 与 `origin/master` 的分支合流。
- 不修改业务代码、测试实现、CI workflow 或 runner。
- 不把每个完成切片拆成独立归档文件。
- 不要求 `docs/plans/` 中的长期路线图、索引或滚动技术债清单使用 todo 归档规则。
- 不删除完成证据；归档是保留历史，不是清理历史。

## 3. 方案比较

### 3.1 方案 A：原地标记完成

完成后只修改 todo 的状态和 checkbox，文件仍留在原位置。

- 优点：改动最小，引用路径不变。
- 缺点：活跃目录和索引仍会发现它；执行前草稿、完成态和历史记录容易混在一起；无法从结构上阻止重复激活。

### 3.2 方案 B：整份 todo 终态归档

单切片完成时更新原 todo；全部切片终结后，把整份文件移入 `docs/todos/archive/`，同步清理活跃入口和未来引用。

- 优点：活跃集合和历史集合边界清楚；保留完整切片演进记录；符合现有仓库归档惯例；能直接防止重复选题。
- 缺点：归档时必须审查引用和索引，不能只移动文件。

### 3.3 方案 C：每轮拆分已完成与剩余文件

每完成一部分切片，就把完成部分迁入历史文件，剩余部分保留为新的活跃 todo。

- 优点：活跃文件始终只含未完成项。
- 缺点：频繁拆分和改名；同一目标态会产生多个事实源；跨切片依赖和审计链容易断裂。

### 3.4 选择

采用方案 B。它以整份 todo 为稳定能力包容器，以切片为执行单位，以归档为 todo 的终态转换，既保持厚切片历史完整，也能从目录和索引层消除陈旧活跃入口。

## 4. 生命周期定义

### 4.1 活跃 todo

活跃 todo 是 `docs/todos/` 下除 `archive/**` 外，仍承载至少一个可执行能力包，或仍被当前入口 / 索引声明为待执行的 todo 文档。子目录中的专题 todo 同样适用；README 和索引只负责发现，不视为 todo。活跃 todo 可以包含已完成切片，但必须准确标识当前剩余项、依赖、优先级和下一候选。

以下情况都表示 todo 仍然活跃：

- 至少一个厚切片尚未完成。
- 任一关键验证为 `FAIL`、`NOT_RUN`、`BLOCKED`、`TIMEOUT` 或 `FLAKY`，且该结果属于不可豁免门禁，或未满足 Playbook 的受控例外。
- 完成实现只存在于未合流分支、其他 worktree 或尚未进入当前分支已验证待交付工作树的远端提交。
- 剩余 P2/P3 项仍被明确列为当前可执行能力包，尚未迁移到新的 owning backlog 或路线图。

### 4.2 切片终态

一个切片只有在以下任一条件成立时进入终态：

1. **完成**：目标态、验收和不可豁免质量门已经满足，结果证据绑定当前事实版本。
2. **取消**：经 `ASSESS` 证明不再需要，并记录决策依据和影响；真正取消不要求虚构后续 owner。
3. **替换**：被新的同级能力包取代，并记录新 owning todo 或计划位置。
4. **迁移**：被移动到另一个 backlog、路线图或 todo，并记录明确路径、优先级和重新激活条件。

影响实现、切片验收或不可豁免门禁的外部阻塞、暂缓、等待凭证、环境不可用或“以后再看”不是终态；非硬性证据缺口只按 Playbook 第 5.1 节的受控例外判断。

### 4.3 Todo 终态

一份 todo 只有同时满足以下条件才进入终态：

- 所有厚切片均处于切片终态。
- 所有被取消事项都有可审计的决策依据和影响；所有被替换或迁移事项都有明确 owning 去向。
- 没有未解释的 checkbox、占位、残余 P0/P1 或隐含后续实现。
- 完成代码和文档已经位于当前分支的已验证待交付工作树；若实现位于其他分支，必须先安全合流并验证组合态。
- 最终验证、CI 等价判断、残余风险和远端同步状态已记录。
- 已无能力包需要该 todo 继续充当活跃执行入口；实际索引清理由归档事务完成。

## 5. 归档事务

Todo 进入终态后，`DELIVER` 必须把以下动作视为同一归档事务：

1. 把 todo 顶部状态更新为已完成、已取消或已迁移。
2. 更新每个切片的终态、完成证据、已知前置提交和验证结果；本轮最终 commit 用“本归档文件所在提交”自指，精确 SHA 在提交后写入交付说明，避免为记录自身 SHA 反复 amend。
3. 确认剩余能力包为零；条件触发的 P2/P3 候选若需保留，先迁入新的 owning backlog 或路线图。
4. 确认目标归档路径不存在；如已存在，先生成身份判断与双方独有证据差异清单，显式消解冲突并禁止覆盖。确认 canonical 归档保留全部独有证据后，再保留原文件名或使用明确消歧名称，将 todo 移入 `docs/todos/archive/`。
5. 从 README、索引、“当前入口”和其他面向未来的执行文档中移除或替换该路径。
6. 全仓扫描旧路径；把命中分类为活跃引用、迁移记录或历史事实。
7. 逐条按语义修复面向当前或未来执行的引用；只有上下文明确属于历史事实的迁移设计、实施计划、提交记录或带日期执行快照才保持原样，不能按文档类型整体豁免。
8. 验证归档文件存在、原活跃路径不存在、活跃索引不再列出该 todo。
9. 运行文档一致性检查和 `git diff --check`，核对 staged ownership。
10. 把归档移动、索引清理、必要引用迁移与本轮已验证实现纳入同一个最终聚焦 commit；若实现早已存在于先前 `HEAD`，则形成独立的归档收口 commit。提交后确认新 `HEAD` 包含完整归档事务，完成说明列出归档路径和历史引用分类。

任一步失败时，归档事务不完整，必须留在 `DELIVER` 修复；不能进入 `NEXT`。

## 6. 融合现有 Playbook 的位置

本设计不增加新状态，也不改变四字段契约数量；通过以下五处融合解决终态收口路由和归档门禁。

### 6.1 状态图与 `ASSESS`

为无需行为实现的稳定记录收口增加 `ASSESS → DELIVER` 路由：当 `ASSESS` 判定当前能力包取消、替换或迁移，且该判断产生必须落盘的稳定记录变更时，直接进入 `DELIVER`，不伪造 `IMPLEMENT` / `VERIFY`。owning todo 仍有其他切片时只更新切片状态和下一入口；该判断使整份 todo 进入终态时才执行归档事务。`BOOTSTRAP` 发现“实现早已完成但 todo 尚未归档”的陈旧入口时，也先由 `ASSESS` 核验证据，再走此路由。

现有 `ASSESS → NEXT` 只适用于 owning todo 仍然活跃且没有待交付稳定记录变更的情况；不得用它绕过终态 todo 的归档事务。

同时增加 `DELIVER → ASSESS` 返回路由：只有终态判断、取消 / 替换 / 迁移依据、owning 去向或陈旧完成证据在交付阶段失效时走该路由；不能把一轮未经实现验证的收口工作错误送入 `VERIFY`。

### 6.2 `1.4 文档与产物职责`

把 todo 职责拆成两类语义：

- 活跃 todo：承载仍需执行的需求、目标态、能力包、依赖、状态和结果证据。
- 归档 todo：只承载已终结工作的历史证据，不参与默认候选聚合。

### 6.3 `2.8 DELIVER`

`DELIVER` 增加两类受控进入条件：一是来自 `VERIFY` 的已验证实现；二是来自 `ASSESS`、无需行为实现但已有充分决策证据的稳定记录收口，包括非终态切片状态更新与终态 todo 归档。后者只能做稳定记录、必要归档和交付工作，并至少完成文档一致性与 staged ownership 检查，不能借此跳过本应执行的实现或质量门。

在必需产物中增加 todo 终态判断和归档决定。在离开门禁中增加：若 todo 已进入终态，归档事务必须完成且新 `HEAD` 同时包含相关实现事实与归档收口；若 todo 仍有未终结切片，只更新当前切片状态和下一入口，不得提前归档。

失败路由按证据来源区分：来自 `VERIFY` 的实现验证证据失效时回到 `VERIFY`；终态判断、决策依据、owning 去向或陈旧完成证据失效时回到 `ASSESS`；路径冲突、索引、引用、staging 或提交归属问题留在 `DELIVER` 修复；只有必需外部条件缺失才进入 `WAIT`。

### 6.4 `2.9 NEXT`

在进入条件和离开门禁中明确：已终结 todo 不得继续留在活跃入口；todo 归档未完成时不能进入 `NEXT`。

### 6.5 `6.1 稳定记录更新`

在 Playbook `6.1` 新增“Todo 收口与归档”小节，作为规则唯一详细归属。`DELIVER` 和 `NEXT` 只引用该小节，不重复十步事务。

## 7. 失败与例外处理

### 7.1 未合流分支

如果实现和完成版 todo 只存在于其他分支、worktree 或远端，当前 todo 不能据此归档。目标模式必须先把实现安全合入当前分支的待交付工作树并完成组合态验证；若实现已由先前提交合入，则以当前 `HEAD` 为事实，若是本轮最后切片，则与归档事务一起进入最终聚焦 commit。缺少外部权限时按实际情况进入 `WAIT`。

### 7.2 外部阻塞

凭证、网络、额度、外部服务或环境阻塞若影响实现、切片验收或不可豁免门禁，todo 不能进入终态，必须保持活跃、记录恢复触发器并进入 `WAIT`。只有 owning 事实源未标记为 blocking / must-pass 的非硬性证据缺口，且完整满足第 5.1 节受控例外时，才允许终态收口；原始 `NOT_RUN`、`BLOCKED` 或 `TIMEOUT` 状态、影响范围、风险、用户接受和恢复条件必须保留，不能改写为 `PASS`。

### 7.3 条件触发 backlog

不再属于当前主线、但未来可能由新证据触发的 P2/P3 项，必须先迁移到明确的 owning backlog 或路线图。只有原 todo 不再承担其执行入口后，原 todo 才能归档。

### 7.4 历史引用

归档后允许单条设计、计划、提交记录和带日期执行快照继续引用原路径或当时文件名，但必须逐条确认上下文明确属于历史事实。任何仍承担当前或未来执行入口的引用都必须更新；历史引用不能被 `BOOTSTRAP` 当作活跃候选。

### 7.5 滚动文档

明确标识为索引、稳定路线图或滚动技术债登记册的文件不按 todo 终态归档。它们必须维护自己的当前 / 历史区分，且不能伪装成单个可完成 todo。

### 7.6 归档路径冲突

如果目标归档路径已经存在，禁止覆盖、静默合并或凭文件时间猜测新旧。目标模式必须比较 todo 身份、内容、提交历史和引用，并列出双方独有的切片状态、验证和提交证据：同一 todo 的重复历史只有在全部独有证据进入 canonical 归档后才能删除重复文件；不同 todo 应选择明确且不冲突的归档文件名，并同步未来引用。冲突未消解前保持在 `DELIVER`。

## 8. 验证设计

### 8.1 静态检查

- Playbook 仍只有原有十个状态、十个状态标题和四十个四字段契约项；只新增必要转换边，不新增状态。
- 新规则的详细定义只存在于 Playbook `6.1`，其他章节只做门禁引用。
- `ASSESS` 的非终态切片记录更新和终态 todo 收口都可达 `DELIVER`，且不会伪造 `IMPLEMENT` / `VERIFY`；只有整份 todo 终态才归档，`ASSESS → NEXT` 不能绕过待交付记录或归档事务。
- `DELIVER` 中失效的终态 / 决策证据可回到 `ASSESS`；实现验证证据才回 `VERIFY`，事务问题留在 `DELIVER`。
- `docs/todos/archive/` 被明确为历史证据集合，不是默认候选源。
- Playbook 不增加具体业务 todo 名称、当前模块清单或易漂移命令。
- `git diff --check` 通过。

### 8.2 行为探针

1. 前五个切片已在 `HEAD`，最后一个切片位于当前分支已验证待交付工作树：实现与归档事务进入同一最终 commit，新 `HEAD` 包含两者后才进入 `NEXT`。
2. 五个切片完成、一个切片未完成：todo 保持活跃，只更新下一入口。
3. 六个切片在远端分支完成但当前分支未合流：不得归档，先合流和验证。
4. 最后一个切片因外部凭证 `BLOCKED` 且影响验收或不可豁免门禁：todo 保持活跃并进入 `WAIT`。
5. 非硬性外部证据为 `BLOCKED` 且完整满足第 5.1 节受控例外：允许收口，但保留原状态、风险、用户接受和恢复条件。
6. 最后一个切片在 `ASSESS` 被有证据地取消：走 `ASSESS → DELIVER` 记录决策并归档，不伪造实现或验证。
7. `BOOTSTRAP` 发现实现早已在 `HEAD`、但 todo 仍活跃：由 `ASSESS` 核验后走 `DELIVER` 补做归档事务。
8. 剩余 P2 已迁入明确 backlog：记录去向后允许归档原 todo。
9. 剩余 P2 只有“以后再看”文字：不得归档。
10. 文件已移动但 README 仍列为当前入口：留在 `DELIVER` 修复。
11. 历史执行快照引用旧路径：按单条语义保留历史引用，不恢复为活跃候选。
12. 归档后 `BOOTSTRAP` 扫描：只聚合活跃 todo，不从 archive 恢复实现。
13. 目标归档路径已存在：禁止覆盖，留在 `DELIVER` 生成双方独有证据清单并显式消歧。
14. 从 `ASSESS` 进入 `DELIVER` 后发现取消依据不足：回到 `ASSESS` 重评，不进入未满足进入条件的 `VERIFY`。
15. 多切片 todo 只取消其中一个切片且必须落盘：走 `ASSESS → DELIVER` 更新记录，但 todo 保持活跃且不归档。

## 9. 验收标准

1. 完成 todo 不再留在活跃目录或当前入口中。
2. 部分完成、受不可豁免外部条件阻塞或未合流的 todo 不会被错误归档；受控例外不会被错误改写成 `PASS`。
3. 被取消的切片有明确决策依据和影响，被替换或迁移的切片有明确终态和 owning 去向。
4. 归档移动、活跃入口清理、未来引用迁移和完成证据构成同一 `DELIVER` 门禁；离开时新 `HEAD` 包含完整事务。
5. `NEXT` 不能绕过未完成的归档事务。
6. 历史引用和完成证据被保留，但 archive 不参与默认选题。
7. 当前 Playbook 的十状态结构、诚实验证语义、厚切片规则和文档职责保持不变。
8. 独立 reviewer 未发现新的 P0/P1 状态机矛盾或归档绕过路径。
9. 同名归档冲突不会造成历史文件被覆盖或两个 todo 被静默合并。

## 10. 当前工作区说明

本设计只定义未来生命周期规则，不把当前分支合流和现有架构 todo 归档混入同一实施。当前 `master` 与 `origin/master` 的架构提交 / Playbook 提交合流，应作为独立、可验证的集成工作处理；合流后的正式完成版 todo 再按新规则归档。
