排序结论：
1. 选择“冲突草稿逐行合并”，因为它是用户最容易遇到的协作恢复场景，并且可以复用现有冲突 diff、草稿和保存流程形成闭环。
2. 完整文档逐行审阅和服务端级合并暂缓，避免本轮变成通用合并编辑器。

切片厚度门禁：
- 入口：保存人工编辑遇到服务端版本冲突后，点击 `对比服务端版本`。
- 动作：用户对草稿新增行点击 `采纳到草稿` 或 `丢弃此行`。
- 处理：系统更新编辑草稿文本，不直接覆盖当前 artifact 或服务端版本。
- 可见结果：diff 中对应新增行消失或进入草稿正文，编辑框内容同步变化。
- 状态承接：用户可继续点击 `保存修改` 走现有保存/冲突机制。
- 失败反馈：本轮只暴露能定位的新增行操作；无法定位时不显示逐行操作。
- 证据：ArtifactPane 组件测试覆盖采纳和丢弃。
- 结论：通过。

## 用户故事

作为正在校准右侧产出物的用户，当保存时发现服务端已有新版本，我可以在冲突对比里只保留自己草稿中的某几行改动，丢弃不需要的行，从而不用在两个版本之间整段复制粘贴。

## 验收条件

1. Given 人工编辑保存遇到 artifact conflict
   When 用户打开 `对比服务端版本`
   Then 草稿新增行旁显示逐行操作，且服务端删除行不显示采纳/丢弃操作。

2. Given 冲突 diff 中存在草稿新增行
   When 用户点击 `丢弃此行`
   Then 编辑草稿删除该行，diff 立即更新。

3. Given 冲突 diff 中存在草稿新增行
   When 用户点击 `采纳到草稿`
   Then 当前编辑草稿以服务端版本为基准并包含该新增行，用户可继续保存。

## 非目标

- 不实现通用三方 merge 算法。
- 不处理删除行或修改行的精确语义合并。
- 不新增后端 patch API。
- 不改变现有 artifact history、服务端更新和冲突检测契约。
