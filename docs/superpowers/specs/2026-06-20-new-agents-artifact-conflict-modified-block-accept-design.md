# New Agents Artifact 冲突修改块采纳设计

## 背景

Artifact 手工编辑支持冲突检测、行级采纳/丢弃和连续新增块处理，但当服务端版本与用户草稿在同一位置各自修改多行时，当前 UI 只把用户草稿识别为新增块。用户必须逐行处理，且逐行采纳会把内容追加到末尾，不能表达“用草稿修改替换服务端同位置修改”的意图。

## 目标

- 在现有 `buildLineDiff` 行级 diff 基础上识别相邻的 `removed* + added*` 修改块。
- 在修改块的草稿新增起始行提供 `采纳修改块` 操作。
- 采纳时以服务端当前版本为基准，将服务端对应删除块原位替换为用户草稿修改块。
- 保留服务端在其他位置的新增或修改，避免用户为采纳局部修改而丢失服务端更新。
- 写入 Artifact 活动轨迹，便于历史面板回看人工合并决策。

## 非目标

- 不引入新的 diff/merge 引擎。
- 不改变 `/api/agent/runs/:runId/artifacts/:stageId` 保存接口。
- 不新增 agent 或 workflow 专属渲染/状态分支。
- 不实现完整三方 merge 冲突解析；本切片只覆盖相邻修改块的采纳。

## 设计

1. 在 `ArtifactPane` 中从 `conflictDraftDiff` 派生修改块：
   - 连续 `removed` 行作为服务端修改块。
   - 后续连续 `added` 行作为用户草稿修改块。
   - 两侧均存在非空行时生成块级操作元数据。
2. 点击 `采纳修改块`：
   - 将 `conflictArtifact.content` 拆成服务端行。
   - 找到首个与服务端修改块完全匹配的连续行序列。
   - 用用户草稿修改块替换该序列。
   - 更新 `editDraft`。
3. 记录 `artifact_merge_block_modified_accepted` 审计事件：
   - 摘要包含服务端块与草稿块的对照标签。

## 验收

- 修改块按钮只出现在相邻 `removed* + added*` 的草稿新增起始行。
- 采纳后草稿内容以服务端当前版本为基准，并原位替换目标修改块。
- 服务端其他位置的内容被保留。
- 活动轨迹记录本次块级修改采纳。
