### 排序结论

1. 选择章节级非重叠改写自动合并，因为 PDF flowchart/timeline/mindmap/pie 已连续补强，本轮应补 Artifact 协作体验中仍可安全推进的高价值缺口。
2. 移动语义暂不选，因为当前 diff 基础不足以可靠区分移动、复制、误删和改写，自动化风险大于收益。
3. Journey 暂不选，因为用户旅程阶段已有结构化 `journey-map` 表格兜底，边际价值低于协作冲突闭环。

### 切片厚度门禁

- 入口：ArtifactPane 人工编辑保存遇到服务端版本冲突。
- 动作：用户点击 `自动合并非重叠变更`。
- 处理：系统基于唯一 Markdown 标题章节判断服务端和草稿是否改写了不同章节，安全时合成新草稿。
- 可见结果：编辑区显示按原章节顺序合并后的内容，同时保留服务端章节改写和用户章节改写。
- 状态承接：用户仍需点击 `保存修改` 走现有冲突检测和服务端保存；自动合并写入本地 Artifact 活动轨迹。
- 失败反馈：同一章节双方都改、重复标题、章节顺序变化或新增/移动章节时不显示自动合并入口，继续走人工对比。
- 证据：先写失败测试，再实现，运行 ArtifactPane 测试、lint/build 和 `git diff --check`。
- 结论：通过。该切片形成完整协作恢复动作链。

## 本轮 Milestone

作为 New Agents 用户，当我在旧版本产出物中改写一个章节，而服务端已经改写了另一个章节时，我可以在保存冲突后通过一个入口自动合并这些非重叠章节改写，从而减少手工逐块处理冲突的成本，同时保留可审计轨迹。

## 范围

- 只处理唯一 Markdown 标题章节。
- 服务端和草稿的标题集合与顺序必须和 base 一致。
- 服务端和草稿改写的章节集合必须不重叠。
- 合并结果按 base 的章节顺序重建。
- 复用现有 `自动合并非重叠变更` 入口和 `artifact_auto_merge_applied` 审计事件。

## 非目标

- 不处理章节移动、重复标题、新增章节或删除章节。
- 不做 Markdown AST/source map。
- 不改变服务端 artifact update API。
- 不自动保存合并结果。

## 验收条件

1. 服务端和草稿分别改写不同章节时，冲突卡片显示 `自动合并非重叠变更`。
2. 点击后编辑草稿包含两边章节改写，并保持原章节顺序。
3. 活动轨迹记录自动合并章节改写。
4. 双方改写同一章节、重复标题或标题顺序变化时不显示自动合并入口。
