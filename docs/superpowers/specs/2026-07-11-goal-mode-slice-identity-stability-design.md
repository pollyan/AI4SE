# Goal Mode 厚切片身份稳定性设计

## 背景与问题

现行 Playbook 已要求“spec、plan、验收、commit 和进度只以厚切片为单位，不设置隐藏内部批次”，但执行过程中仍可能把同一厚切片的内部 TDD 步骤命名为 `A/B/C`、phase、batch 或 checkpoint，并进一步把这些内部步骤当作独立进度、验收或提交边界。问题不是缺少厚切片原则，而是缺少贯穿状态转换的可操作身份不变量。

## 目标与非目标

目标是让用户已经确认、且当前事实仍支持的厚切片序列稳定执行：厚切片的 ID、名称、用户任务、七项门禁和交付边界在执行期间保持一致；内部实现步骤只能服务当前厚切片，不能获得子切片语义。

本次不重新发明切片方法，不改变 QS-01 至 QS-08 的业务范围，不禁止 TDD 小步、实现清单或子智能体任务，也不阻止新 P0/P1、目标失效或自然业务边界变化触发正式重评估。

## 方案比较

1. 只依赖现有禁止隐藏批次的文字。改动最小，但现有事件已经证明它不足以约束执行中的命名、进度和提交行为。
2. 再增加一句“禁止 A/B 子切片”。可提高可见性，但容易被 `phase`、`batch`、`checkpoint` 或其他名字绕过，也没有规定真正需要拆分时的合法路线。
3. 建立“厚切片身份稳定性门禁”。推荐此方案：冻结切片身份，把所有内部清单明确标记为非切片，在状态转换、进度、验收、commit 和交付前重复检查；真正需要拆分时只能回到 `ASSESS`，产生具有完整七项门禁的同级厚切片并记录替代关系。

## 设计

### 身份基线

`MILESTONE` 通过时确定临时厚切片身份基线，至少包含 ID、名称、完整用户任务、纳入/排除边界、七项门禁、依赖、验收证据和单一交付边界。完成 brainstorming 后，`DESIGN` 把临时基线持久化到当前中文 spec 的固定章节，未持久化不得进入 `PLAN`；用户已确认的有序切片序列以 owning todo 对应段落作为唯一顺序基线。plan、进度与交付记录只链接基线，不复制改写。

### 执行不变量

- 计划可以包含多个 RED/GREEN 步骤、测试、helper 和子智能体任务，但必须标识为“内部实现步骤（非切片）”。
- `A/B/C`、小数编号、phase、batch、wave、checkpoint、task 等内部标签不得用于宣称新的切片、进度单位、验收边界、独立 commit、push 或交付。
- 当前切片只有整体通过七项门禁、验证和交付规则后才能计为完成；内部步骤完成不能提高“已完成厚切片数”。
- 当前厚切片默认形成一个聚焦 commit。实现中的临时本地 checkpoint 不能成为交付事实；若已经产生，必须在交付前整理为当前厚切片的单一提交，且不能把未完成切片推送为完成。
- 当前切片内发现并修复的 bug 默认并入当前切片，不能仅改称 bugfix 后独立提交；只有重新进入 `ASSESS` 并证明它是拥有完整七项门禁的同级能力包，才可独立交付。

### 合法改界路线

只有用户目标变化、新 P0/P1、目标失效、七项门禁失真或发现自然业务边界时，才能停止当前路线并回到 `ASSESS`。工作量、文件数、上下文、测试耗时、切换窗口或 Agent、保存 checkpoint 和提交便利都不是自然业务边界。重新划分必须生成同级、各自拥有完整用户任务和七项门禁的厚切片，记录原切片的取消、替代或迁移关系以及剩余工作的 owner；不得用 `QS-03A/QS-03B` 一类名称延续私有碎片。

### 状态转换检查

从 `MILESTONE` 起，每次离开 `DESIGN`、`PLAN`、`IMPLEMENT`、`VERIFY`、`DELIVER` 和 `NEXT` 前检查：身份和顺序是否仍与基线一致；内部标签是否被赋予切片语义；进度、验收、commit 和交付是否仍以完整厚切片记账；剩余工作是否仍映射到原七项门禁；如需改界是否已经回到 `ASSESS`。

## 现有 QS-01 至 QS-08 评估

八个能力包均以独立用户任务或工程信任闭环命名，分别拥有风险、能力增量、目标态、纳入/排除范围、架构边界、可观察证据、依赖和相邻切片边界。当前没有证据表明其中任何一个必须私自拆薄：QS-01 至 QS-06 依次关闭 P0 前置链，QS-07 形成“选择、执行、诊断测试”的完整反馈闭环，QS-08 形成“唯一事实源与漂移治理”的完整静态治理闭环。因而序列保持 `QS-01 → … → QS-08`；只有 Playbook 定义的强制改道事实出现时才重新评估。

## 验证

- 文档检查确认 Playbook 同时包含身份基线、禁止伪装子切片、合法改界路线和状态转换检查。
- 当前 QS-03 implementation plan 不再出现 `4A/4B/4C`，所有 task 明确为内部实现步骤（非切片），交付仍只有一个 QS-03 验收与 commit。
- `git diff --check` 通过，新增规则不与自主执行、七项门禁、状态机和提交规则冲突。
