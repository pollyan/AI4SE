# New Agents Artifact 冲突非重叠插入自动合并设计

## 背景

Artifact 手工保存冲突已经支持刷新服务端版本、查看 diff、逐行采纳/丢弃、新增块采纳/丢弃、修改块采纳和保留服务端。但当服务端和用户草稿都基于同一旧版本，只是在不同或相邻位置补充新内容时，用户仍需要手工逐行处理。

## 目标

- 在保存冲突中检测“双方都只是相对旧版本插入内容”的安全场景。
- 为该场景提供 `自动合并非重叠变更` 操作。
- 自动合并后编辑草稿同时保留服务端补充和用户草稿补充。
- 写入 Artifact 活动轨迹，记录自动合并决策。
- 保留现有手工冲突处理能力，不改变服务端保存 API。

## 非目标

- 不实现完整三方 merge 引擎。
- 不自动处理删除、改写、重复锚点歧义或复杂移动。
- 不在冲突发生时自动修改草稿；必须由用户点击按钮触发。
- 不新增 workflow/agent 专属分支。

## 设计

1. 以当前 `artifactContent` 作为旧版本基线，`conflictArtifact.content` 作为服务端较新版本，`editDraft` 作为用户草稿。
2. 对服务端版本和用户草稿分别执行“插入段收集”：
   - 旧版本所有行必须能按顺序在目标版本中找到。
   - 旧版本行之间多出来的内容视为插入段。
   - 如果任意旧版本行缺失，视为存在删除或改写，不提供自动合并。
3. 合并时按旧版本行顺序输出：
   - 当前 gap 的服务端插入段。
   - 当前 gap 中用户草稿独有插入段。
   - 对应旧版本行。
4. 如果合并结果没有引入任何草稿独有内容，或等同服务端版本，不显示按钮。
5. 点击按钮后更新 `editDraft`，并记录 `artifact_auto_merge_applied` 活动事件。

## 验收

- 当服务端和草稿都只插入内容时，冲突横幅显示 `自动合并非重叠变更`。
- 点击后编辑草稿包含服务端插入、用户插入和原有基线内容。
- 活动轨迹记录自动合并。
- 聚焦测试先失败后通过。
