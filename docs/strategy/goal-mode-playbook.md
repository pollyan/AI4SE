# AI4SE 目标模式运行手册

本文是 Codex / AI Coding Agent 在目标模式下持续演进 AI4SE 的执行入口。它保留每轮必须遵守的规则；CGA 模板和 CI 等价验证分别下沉到专题附录，避免把长模板和命令矩阵塞进主流程。子智能体调度属于目标模式执行循环本身，直接收敛在本文中维护。

## 1. 定位

目标模式的职责不是复述 AI4SE 的仓库结构或机械执行任务清单，而是基于当前代码、文档、测试和用户最新目标，持续选择一个最有价值、可验收、风险可控的用户故事推进。仓库结构、架构事实、API 契约、测试命令和编码规范分别以 `AGENTS.md`、`docs/index.md`、`docs/ARCHITECTURE.md`、`docs/api-contracts.md`、`docs/TESTING.md`、`docs/CODING_STANDARDS.md` 和 `docs/DESIGN_PRINCIPLES.md` 为准。

目标模式由以下文件共同驱动：

- `AGENTS.md`：仓库级硬约束。本手册不重复其中的编码、架构、测试和提交细则。
- `docs/index.md`：项目知识库入口，说明模块、架构、API、数据、部署和深层文档导航。
- `docs/strategy/goal-mode-playbook.md`：目标模式通用运行入口，也就是本文档。
- `docs/strategy/goal-mode-cga-template.md`：CGA 模板、切片厚度门禁和常见过薄切片处理。
- `docs/strategy/goal-mode-ci-verification.md`：CI 等价验证、远端失败复盘和提交前验证表。
- `docs/todos/`：长期演进 todo，记录跨目标模式持续消化的高优先级产品、工程、测试和评判事项。
- `docs/plans/`：专项计划和专项目标模式规则。只有当本轮目标指向对应主题时，才展开相关计划。
- 本轮 spec / plan / 验证记录默认压缩写入相关 `docs/todos/*.md` 或正式 `docs/plans/*.md`，避免长期保留大量过程文件。

提示词应保持短小，只引用稳定文件，不重复本手册、`AGENTS.md` 或深层需求文档中的细则。

## 2. 目标与授权边界

长期目标是让 AI4SE 保持为一个可本地部署、可演示、可持续演进的 AI for Software Engineering 工具集。具体模块职责不在本手册维护；每轮按启动协议读取稳定事实源，并在 CGA 中说明本轮实际消费了哪些模块事实。

如果用户没有指定本轮目标，默认从 `docs/todos/`、当前失败证据、活跃计划和相关 backlog 中选择最高优先级、尚未完成、当前上下文可推进的工作项；此时必须先做完整 Current State Gap Analysis，不能直接套用上一轮结论或固定清单。

如果用户指定了本轮主要目标，也必须快速扫描 `docs/todos/` 中的 P0/P1 条目。若存在与当前目标同向、低冲突、可形成纵向切片的高优先级 todo，应在 CGA 中纳入候选；若本轮不消化，必须说明暂缓原因和后续去向。

如果用户已经确认了目标轮次、阶段顺序和验收边界，并且这些内容已经写入 `docs/todos/`、专项计划或上一轮执行记录，后续轮次不需要重复完整候选排序型 CGA。此时使用轻量“目标承接检查”：确认当前轮次仍是已声明顺序中的下一项、没有新的 P0/P1 冲突或失败证据需要改道、工作区安全、切片边界和验收证据仍清楚，然后直接承接进入需求细化、实施计划和 TDD。若出现目标变更、新高优先级冲突、代码事实与计划冲突、需要拆分或合并轮次、或用户要求重新评估，则恢复完整 CGA。

目标承接检查不是继续推进的豁免。若上一轮或当前验证留下未关闭的质量门失败、LLM judge 低于 80 分、阻断测试失败、用户明确指出的未处理缺陷或安全 / 契约风险，该事项自动成为当前轮 P0 改道条件。除非阻塞原因是权限、凭证、额度、外部服务或环境不可用并已记录暂缓风险，否则不得进入后续目标轮次，也不得把它降级为普通残余风险。

目标模式运行中收到用户新的纠错、质疑或补充反馈时，最新用户消息优先于当前 plan。主 Agent 必须在下一个安全检查点暂停当前切片推进，先判断该反馈是否指向未关闭质量门、遗漏需求、错误结论、子智能体漏用或验证证据不足；若是，必须把它写入当前轮改道条件或收尾阻塞项，并先修复 / 重跑 / 记录环境阻塞，不能继续沿原计划推进。若判断不是改道条件，也要在 spec、todo 或收尾说明中写明取舍理由。

技术债、New Agents 专项演进、部署治理、评判体系等都属于可被目标模式处理的一类需求；只有当用户目标、todo 或 CGA 候选指向这些主题时，才读取对应专项计划，例如 `docs/plans/tech-debt.md`。

目标模式可以在符合 `AGENTS.md`、本手册、相关深层文档和当前代码事实的前提下，自主完成上下文读取、gap 排序、spec、plan、实现、验证和记录更新。不要把用户确认作为常规节点；普通澄清由 Agent 基于文档、代码、测试和合理 assumptions 自行回答，并在 spec 或收尾说明中记录。

只有以下情况才停止并向用户汇报：

- 需要用户提供权限、凭证、外部服务访问、浏览器登录态或仓库访问。
- 当前任务需要联网安装依赖、访问模型供应商或调用外部 API，而环境未授权。
- 连续阻塞，且无法通过当前文档、代码、测试或合理 assumptions 推进。
- 用户最新消息明确要求暂停、只讨论、不改代码或等待确认。
- 当前架构、文档事实或用户目标之间存在无法自行裁决的根本冲突。

## 3. 启动协议

每轮目标模式启动后，先执行启动协议：

1. 读取 `AGENTS.md`。
2. 读取本手册。
3. 读取 `docs/index.md`，获取当前项目结构和深层文档导航。
4. 检查 `git status --short`。如果存在未提交变更，必须判断它们是用户改动、上一轮遗留还是当前轮输入；不得覆盖、回滚或格式化不属于本轮的变更。若未提交 diff 很大或包含大量删除 / 生成文件，还必须补充 `git status -sb`、`git diff --shortstat` 和按目录归类的 ownership 判断，区分“本轮可提交 diff”和“需保护的无关脏变更”。
5. 按任务方向读取相关事实源：架构、API、测试、编码规范、设计原则、todo、专项计划、已有 spec / plan、当前代码和测试。
6. 判断是否需要读取专题附录：CGA 模板见 `goal-mode-cga-template.md`，CI / 提交前验证见 `goal-mode-ci-verification.md`。
7. 检查未关闭质量门和用户最新反馈。任何 LLM judge 低于 80 分、阻断测试失败或用户明确指出的遗漏，都必须先作为改道条件进入本轮判断；如果反馈发生在本轮运行中，也必须先中断当前推进并更新改道判断。
8. 第一个实质性工作产物必须是完整 Current State Gap Analysis；若本轮是已确认目标轮次的连续执行，则第一个实质性工作产物必须是目标承接检查。该产物必须包含子智能体 / 旁路审查决策：可并行读取、审查、验证或互不重叠实现且环境提供子智能体工具时，必须实际尝试分发；不分发时写清原因。若分发失败，按本文“并行协作与子智能体”规则降级并记录。

不要先写 implementation plan、先改代码、先跑固定 todo 清单，或直接沿用上一轮记忆。已确认目标轮次也必须先做承接检查，不能在不验证当前事实源和工作区状态的情况下机械继续。

### 3.1 单工作区执行原则

目标模式默认不再创建新的 git worktree。历史实践证明，为每个切片创建独立 worktree 会导致大量中间分支、重复提交、最终集成成本和清理风险；目标模式应优先减少并发写入面，而不是通过大量 worktree 扩大后续合流面。

启动协议完成 `git status --short` 后，必须执行单工作区安全检查：

1. 记录当前分支、`git status --short` 和用户已有未提交改动；不得覆盖、回滚或格式化不属于本轮的文件。
2. 默认在当前工作区串行推进一个 milestone。若当前工作区已有用户改动，优先选择不触碰这些文件的切片；如必须触碰同一文件，先说明冲突和保护策略。
3. 不得自动创建 `.worktrees/` 下的新 worktree，不得把每个候选、每个子任务或每个 worker 拆成独立 worktree。
4. 只读探索可以并行；任何可写实现默认由主 Agent 串行完成，或由子智能体在明确的文件所有权范围内提出 patch / 建议，由主 Agent 统一落地。
5. 只有用户当轮明确要求隔离执行、当前工作区无法安全承载必要修改，或仓库维护任务本身就是清理/合并既有 worktree 时，才允许临时使用已有或新建 worktree。使用原因、路径、分支、清理计划必须写入 CGA 和收尾说明。
6. 如果确需临时 worktree，必须在本轮结束前合回、删除或明确标记保留原因；不得留下未记录的中间 worktree 或未合并分支作为后续目标模式负担。

### 3.2 并行协作与子智能体

子智能体是目标模式的执行加速和旁路审查手段，不是独立决策者。主 Agent 始终负责选题、边界、最终集成、验证、提交和完成声明。子智能体输出只能作为工程输入；任何“已修复”“测试通过”“可合并”的结论，都必须由主 Agent 重新检查 diff 和验证证据后才能对用户声明。

每轮 CGA、目标承接检查或收尾说明必须记录：当前是否有可用子智能体工具、是否存在可并行事项、实际派发的 agent id / 范围 / 状态、失败降级情况、不派发原因、本轮允许写入路径和不触碰的既有脏文件。

默认触发点：

- CGA 阶段存在 2 个以上独立候选、多个事实源可并行读取、跨 workflow 契约同步、LLM judge 低分、浏览器级证据审查、文档规则审查、禁止模式扫描，或工作区存在大规模未提交 diff / staging ownership 风险时，优先分发只读 explorer、reviewer 或 verification。
- implementation plan 已明确且任务能拆成互不重叠文件集合时，可以分发 worker；worker 必须有明确文件所有权、测试命令和禁止回滚他人改动的约束。worker 不得自行 commit 或 push。
- 当前工作区有大量未提交改动时，优先只派发只读 explorer / verification；只有写入范围能精确隔离时才派发 worker。

分发规则：

- 子智能体运行期间，主 Agent 继续处理不重叠工作，不空等。
- 多个 worker 的写入范围必须互不重叠；高冲突文件，例如大型 store、`ArtifactPane.tsx`、共享 todo、playbook 和全局配置，默认由主 Agent 统一接线。
- 可写 worker 返回后，主 Agent 必须先读最终说明，再用 `git status --short` 和 `git diff -- <允许路径>` 核对是否越界，随后运行该切片最小验证。
- 只读 explorer / reviewer 的结论必须进入 spec、plan、todo 或收尾说明，不能只停留在对话通知里。

失败降级：

- 子智能体分发失败包括工具不可用、stream disconnected、权限错误、超时未返回、或返回 `BLOCKED` 且无法通过更小范围推进。
- 第一次失败时，记录 agent id、任务目标和失败类型；如果任务仍有并行价值，用更小、更具体、优先只读的任务重试一次。
- 第二次仍失败时停止重试，主 Agent 顺序完成原本分发的审查或验证，并在本轮记录中标注“子智能体环境降级”。
- 如果失败发生在可写 worker，主 Agent 不得假设 worker 有任何有效改动，必须重新检查 `git status --short` 和相关 diff。

## 4. 事实源与冲突优先级

每轮判断都必须基于当前文件，而不是上一轮记忆。

事实源默认顺序：

1. `AGENTS.md`
2. `docs/strategy/goal-mode-playbook.md`
3. `docs/index.md`
4. `docs/ARCHITECTURE.md`
5. `docs/api-contracts.md`
6. `docs/TESTING.md`
7. `docs/CODING_STANDARDS.md`
8. `docs/DESIGN_PRINCIPLES.md`
9. `docs/integration-architecture.md`
10. `docs/component-inventory.md`
11. `docs/todos/`
12. 与本轮相关的 `docs/todos/refactor/README.md` 当前入口、`docs/plans/`、测试需求、当前代码、测试、脚本和 git 历史。

如果某一轮只涉及特定模块，可以按需缩小读取范围，但必须在 CGA 中说明实际读取了哪些关键文件，以及哪些事实源因无关而未展开。

冲突裁决：

- 用户最新明确要求优先于历史文档。
- 当前代码和当前测试优先于过期文档。
- `AGENTS.md` 和本手册优先于普通计划文档。
- `docs/TESTING.md`、`docs/CODING_STANDARDS.md`、`docs/DESIGN_PRINCIPLES.md` 是工程护栏，不应被局部实现便利覆盖。
- 专项计划和专项规则只对其声明的目标范围生效；它们不能替代本手册的通用启动协议，但可以补充对应主题的执行规则。
- 归档 todo、历史计划和旧专项模板只能作为证据来源；如果其中的提交、用户确认、优先级或验证口径与本手册冲突，按本手册执行，并在 CGA 或收尾说明中记录取舍。

发现文档与代码冲突时，不要默默选择一边。必须在 spec、plan 或收尾说明中写清楚冲突、取舍和后续处理。

## 5. 每轮循环

每轮按以下顺序执行：

1. 重新读取事实源和相关代码，不依赖上一轮记忆。
2. 检查 git 状态并保护用户已有改动。
3. 执行完整 Current State Gap Analysis，或在已确认目标轮次场景执行目标承接检查。完整 CGA 要把 todo 子项、用户反馈、测试失败和技术缺口聚合成候选能力包，列出排序依据、未选候选去向、边界和验收证据；承接检查要确认已声明轮次仍有效、没有新的改道条件、边界和验收仍成立。CGA 与承接检查模板见 `docs/strategy/goal-mode-cga-template.md`。该步骤必须显式记录质量门状态和子智能体 / 旁路审查决策。
4. 选择或承接一个且只选择或承接一个用户故事（user story）。用户故事必须是用户可感知的完整能力包，或能独立降低后续演进风险的工程信任闭环。不得把单个控件、字段、helper、parser 分支、三方 merge 场景或单条测试当作独立用户故事，除非它本身解除阻断、风险或证据失真。
5. 选定用户故事后，用 Superpowers `brainstorming` 流程细化需求，再写中文 spec。目标模式的自动执行授权把用户审批节点改为 Agent 基于事实源的自问自答式裁决，但不能跳过上下文探索、澄清、方案比较、设计和测试思考。
6. 写中文 implementation plan，拆成本轮可执行步骤，并标注预计验证和提交边界。
7. 代码或行为变更按 `AGENTS.md` 和 Superpowers `test-driven-development` 技能执行；本手册不重复其具体流程。纯文档变更使用本节的文档核对清单。
8. 运行与本轮改动匹配的最小验证；触及共享行为、跨层协议或用户主路径时扩大验证范围。
9. 更新必要的文档记录，例如 `docs/todos/`、相关专项计划、spec / plan 或测试策略说明。
10. 完成型代码用户故事在进入提交或 push 前必须重跑仓库全量本地自动化，默认命令为 `./scripts/test/test-local.sh all`；这一步必须发生在最终代码和文档落定之后、`git commit` 之前。该命令必须让严重 lint、测试、构建或 E2E 失败反映为非零退出；如果脚本行为与此不一致，必须先修脚本或额外运行对应命令并在收尾说明中记录。只有纯文档变更、小范围无代码变更、环境权限阻塞或用户明确要求跳过时才允许例外，并必须在收尾说明中写明原因和风险。
11. 如果本轮验证启用、要求或引用了 LLM judge，默认通过线为 80 分。低于 80 分不得视为质量通过，也不得用“确定性测试通过”替代质量门禁；必须立即停止推进后续目标轮次，把该失败转为当前 P0 修复故事，记录 judge 反馈、分析产出缺口，修复 prompt、contract、renderer、测试数据或证据链后重跑。差距分析至少写清：当前分数、失败维度、期望产物、实际产物缺口、修复位置、重跑命令和新分数 / 阻塞原因。若存在可并行的差距分析、契约复核或测试证据审查，应分发只读 explorer / reviewer 子智能体；若不分发，必须记录原因。只有环境权限、凭证、额度或外部服务阻塞时才允许暂缓重跑，并必须在收尾说明中写明阻塞原因和风险。
12. 完成型用户故事在进入提交或 push 前必须做 CI 等价验证映射，详见 `docs/strategy/goal-mode-ci-verification.md`。全量本地自动化不能替代按改动风险选择的聚焦红绿测试；聚焦验证和全量验证都要报告。
13. 目标模式下，完成一个通过验证的完整用户故事、独立工程信任闭环或其他可独立说明价值的切片后，必须先批量运行该切片对应的聚焦验证、全量自动化或明确记录的验证例外，再形成聚焦 commit；如果 GitHub 远端可用、当前分支允许推送且用户没有明确要求暂缓，commit 后应及时 push 到 GitHub。进入下一切片前不得把多个独立价值点或大体量 diff 长期缓存在本地未提交状态。工作区存在无关大 diff 时，不能因此跳过提交；只要本轮改动能被明确列出并安全 staged，就必须只 stage 本轮路径、提交并推送。
14. commit 前必须做提交归属检查：运行并记录 `git status -sb`、`git diff --shortstat`、`git diff --cached --name-only`，必要时补充 `git diff --name-status` 的目录级摘要。若未提交 diff 中存在大量无关删除、生成包、测试结果或用户改动，收尾说明必须明确它们未被 staged，并报告本轮 staged 文件列表。
15. 只有用户明确要求不要提交或不要 push、当前只是临时探索、验证未完成、存在未裁决风险、本轮改动无法从无关脏改动中安全拆分、缺少远端权限 / 凭证或外部服务不可用时，才可以暂缓 commit / push，并必须在收尾说明中写明原因、风险和下一次提交 / push 触发条件。无关脏改动本身不是暂缓理由；只有无法安全拆分时才构成暂缓理由。
16. 收尾说明必须列出改动、验证命令、未运行验证及原因、CI 等价覆盖判断、子智能体 / 旁路审查的派发、失败降级或跳过原因、LLM judge 分数与低于 80 分时的修复 / 暂缓记录、残余风险、提交 / push 状态、`HEAD` 与 GitHub 远端同步状态、当前未提交 diff 的归属摘要和下一步候选。

## 6. 用户故事粒度门禁

目标模式默认切片单位是“完整用户故事”，不是“代码改动集合”。一个合格用户故事应能回答：

- 用户或调用方从哪里开始。
- 用户要完成什么真实动作。
- 系统完成什么核心处理。
- 成功后用户能看到、保存、恢复、下游消费或评判什么结果。
- 失败时如何显式停止、解释原因并给出下一步。
- 用什么测试、命令、截图、日志或 artifact 证明。
- 完成后能否用一句“用户现在可以...”或“调用方现在可以...”说清楚新增能力。

## 7. Current State Gap Analysis

CGA 的目标是基于最新代码、测试、文档和用户目标，判断下一步最有价值、最可验收、最少破坏当前契约的工作包。

CGA 不是把 backlog 拆得更细的任务分拣器，而是把分散缺口重新组装成可交付能力包的决策过程。默认先问“用户要完成哪段真实任务”，再决定需要一起纳入哪些后端契约、前端入口、状态承接、错误恢复、下游消费和验证证据。只有在完成这一步之后，才能比较候选优先级。

每轮 CGA 必须包含：

- 事实源快照：本轮实际读取了哪些关键文件。
- 能力包聚合：先把 todo 子项、用户反馈、测试失败和技术缺口归并到同一用户场景或工程信任闭环下，再形成候选；不得从单个子字段、单按钮、单 endpoint 或单测试直接进入 milestone 选择。CGA 的最小候选单位必须是“完整用户功能厚切片”：用户从一个真实入口开始，经过必要动作、系统处理、可见反馈、状态承接和失败恢复后，能完成一段有独立价值的功能闭环。
- 候选 gap：至少 2 个候选能力包；如果确实只有 1 个，也要说明为什么其他缺口无法在本轮形成能力包。
- 排序依据：为什么选中的 gap 比其他候选更适合作为本轮 milestone。
- 用户功能包边界：明确本轮代表的完整用户功能是什么，包含哪些同一用户意图下的相邻缺口；如果排除相邻缺口，必须解释为什么它属于不同用户意图、不同风险面或不同验证闭环。不能因为实现方便，把同一用户动作链中的后端契约、前端入口、状态保存、错误提示、下游消费或验证证据拆成连续多轮目标模式。
- 厚切片合并结论：明确哪些原本看似独立的小缺口被合并为同一完整用户功能；如果仍然只选择一个局部点，必须写出阻断、风险或证据失真的受控例外理由。
- 切片准入判断：说明选中 milestone 是否值得消耗完整 Superpowers 流程成本，是否形成用户可感知的完整动作链，是否能一次性消化同一用户场景下的相邻小缺口，以及完成后用户可观察能力相对当前状态的实际增量。若一个候选只能带来“局部更好”而不能让用户多完成一段真实任务，默认必须继续向外聚合，直到形成完整用户功能包，或改选其他候选。
- 切片厚度门禁：明确本轮是否通过“入口、动作、系统处理、可见结果、状态承接、失败反馈、证据”检查；如果缺失其中关键环节，必须说明如何扩大边界，或为什么这是受控工程信任闭环例外。
- 未选候选去向：保留为下一轮候选、写入来源 backlog / 专项计划、归入更大工作包、标记 blocked / wont-do，或明确暂不处理。
- 验收口径：每个候选都要说明可以被什么证据验证。
- 边界判断：哪些内容进入本轮，哪些内容不进入本轮。

## 8. Milestone 选择与粒度

AI4SE 不维护固定路线图。目标模式选题时，应同时考虑用户最新目标、当前代码事实、todo / backlog 优先级、测试可行性和用户可观察价值。

Superpowers 流程成本较高，因此 milestone 粒度默认应偏厚。一次目标模式切片应尽量覆盖同一用户场景下连续相邻的小缺口，形成一个用户可感知的能力包；不要为了快速推进进度，把同一能力链拆成多个“按钮级”“字段级”“helper 级”切片。默认情况下，一个 milestone 至少应能让用户完成一段新的真实操作，或让调用方获得一个可复用、可验证、会改变后续决策的能力。

目标模式的默认切片单位是“完整用户功能厚切片”，不是“代码改动集合”。完整用户功能厚切片必须围绕一个真实用户目标命名，例如“生产环境可视化稳定呈现闭环”“模型供应商异常自检与修复闭环”“Artifact 编辑与生成并发承接闭环”。它应同时包含这个目标下必要的入口、操作、系统处理、可见结果、状态承接、失败恢复和证据。实现时可以按多个 RED / GREEN 小步推进，但 CGA、spec、plan、验收和 commit 边界必须围绕完整功能包组织。

厚切片的默认目标不是“改动足够多”，而是“完成后用户或调用方获得一段完整可用的新能力”。如果某个候选只能让整体进度增加 0.x% 或只能被描述为内部准备工作，CGA 必须先尝试扩大边界，把同一动作链中的后端契约、服务调用、前端状态、可见反馈、错误处理、测试证据和必要文档合并进同一 milestone。只有当扩大后会跨越不相关用户场景、显著增加破坏面，或当前候选本身解除阻断 / 风险 / 证据失真时，才允许保持较小边界。不能为了降低单轮工作量而把可连续交付的同一用户场景拆成多个目标模式轮次。

如果候选 milestone 只能用内部算法、单个渲染分支、单个三方 merge 场景、单个错误分类或单个结构化字段来命名，默认判定为过薄。CGA 必须把它上升到用户功能包命名，例如从“同章节段落插入与改写自动合并”上升为“Artifact 冲突处理收尾能力包”，并把安全自动处理、无法自动处理时的解释、用户手工承接、状态恢复和验证证据一起纳入边界。实现内部可以按多个小 RED / GREEN 步骤递进，但 CGA、spec、plan、验收和 commit 边界应围绕完整用户功能包组织。

如果一个候选看起来只剩“最后一点”，仍不能默认作为薄切片进入实现。CGA 必须先问：这“一点”是否只是某个用户功能闭环的缺失环节？如果是，必须把闭环中的相邻缺口一起纳入；如果不是，必须说明它解除的阻断或证据风险。不要用“剩余工作不多”“顺手补一下”“推进百分比”作为切片成立理由。

同一用户意图下的相邻缺口不得默认留给下一轮。例如用户要“编辑后的 Artifact 能可靠承接后续生成”，自动合并一个安全场景、阻止一个不安全场景、解释为什么无法合并、提供用户可操作的承接路径、恢复后状态一致，属于同一功能包的不同必要环节。只有当相邻缺口需要不同入口、不同角色、不同数据契约、不同服务依赖，或会显著扩大回归面时，才允许拆出，并且必须在 CGA 中写出拆分理由和后续去向。

milestone 选择必须按“能力包 -> 用户动作链 -> 验收证据 -> 实现边界”的顺序收敛，而不是按“还有哪些小 todo -> 本轮挑一个小 todo”的顺序推进。todo 可以保留细粒度缺口，方便持续追踪；CGA 不能把这些细粒度缺口原样当作本轮 milestone。若一个 todo 子项必须依附另一个子项用户才看得见价值，应在同一切片中合并；若合并后工作量过大，应改选另一个可独立交付的能力包，而不是把半条链路先做掉。

厚切片强制规则：

1. **从用户完整任务命名**：milestone 名称必须描述一个完整用户功能或工程信任闭环，而不是实现技术点。例如写“Artifact 编辑与生成并发承接闭环”，不要写“补一个段落 merge 分支”。
2. **按用户动作链扩边**：如果同一用户目标需要入口、操作、服务处理、状态保存、错误反馈和后续消费共同成立，CGA 默认把这些相邻缺口并入同一切片；不能因为它们分布在不同文件、组件、端点或测试中就拆成多轮。
3. **按自然业务边界拆分**：只有当扩大后跨越不同用户目标、不同入口、不同角色、不同数据契约或显著不同风险面时，才允许拆分。拆分理由必须写入 CGA，并给被排除部分明确去向。
4. **用小步实现厚切片，不设内部批次**：厚切片不要求一次写完所有代码，但目标模式只有“切片”这一层计划单位。spec、plan、验收、commit 和进度汇报都必须围绕同一个切片；实现过程中可以有多个 RED / GREEN 小步、多个 helper、多个测试和多个子智能体任务，但不得再设“内部批次”“子切片”或字母后缀轮次。若一个切片过大，必须在计划层面拆成多个同级切片。
5. **禁止用进度百分比反推薄切片**：如果一个候选只能把总体进度从 98% 推到 98.x%，但不能让用户多完成一段真实任务，默认视为切片选择错误。应重新聚合剩余缺口，或把它归入更大的能力包。
6. **最后一点也要归包**：当 backlog 看似只剩某个小功能点时，CGA 必须判断它是哪个完整用户功能的缺失环节。若它不能独立成立，就合并到同一用户功能包；若它只是历史残留且不再有用户价值，应标记完成、降级或移出当前目标，而不是继续消耗完整目标模式流程。
7. **受控工程例外要可复用**：底层工程信任闭环可以作为 milestone，但必须让调用方、后续目标模式或生产排障获得可复用能力，例如协议校验、失败显式化、证据采集或部署安全护栏。不能停在只对某个内部 helper 有意义的改动。

### 8.1 切片厚度门禁

CGA 选择 milestone 前必须执行切片厚度门禁。门禁不是按改动文件数量判断，而是按完成后是否形成真实能力判断。默认应先寻找能够一次性交付完整用户功能的厚切片；只有当用户功能包过大且可以按自然业务边界拆分时，才允许拆成多个 milestone。自然业务边界必须表现为不同用户目标、不同入口、不同下游承接或不同风险面，而不是不同文件、不同组件、不同函数或不同测试。

默认通过标准：

- 有明确用户入口或调用方入口。
- 有清晰的用户目标和完成状态，而不是只说明要改哪个模块。
- 有一个用户或调用方会主动触发的真实动作，而不是单纯内部预备工作。
- 系统完成可描述的核心处理，并产生可观察结果。
- 结果能被保存、恢复、下游消费、评判或作为后续证据承接。
- 失败路径能显式停止并说明原因，不靠 silent fallback。
- 验证证据覆盖入口、处理、结果和关键失败路径。
- 完成后能用一句“用户现在可以……”或“调用方现在可以……”说清楚新增能力。
- 同一用户动作链中没有明显被推迟到下一轮的必要环节；如果有，必须解释技术阻断或风险边界。

默认不通过的信号：

- 用户故事名称只能写成“新增字段”“补一个 endpoint”“修一个 parser”“调整一个按钮”“补一条测试”。
- 验收条件只验证内部函数，不验证 API、页面、工作流、下游消费或可见反馈。
- 当前切片显然只是某个用户动作链的前半段，而后半段没有技术阻断却被留到下一轮。
- 完成后只能汇报百分比增加 0.x%，但说不清用户多完成了什么。

不通过时，先尝试扩大边界，把同一用户场景下相邻的后端契约、前端服务、页面交互、状态持久化、错误反馈、下游消费和验证证据并入同一用户故事。只有扩大后会跨越不相关用户场景、显著增加破坏面，或当前候选本身解除 silent fallback、安全风险、证据失真或 CI 阻断时，才允许保持较小边界。

详细门禁、CGA 模板、过薄切片示例和能力包进度口径见 `docs/strategy/goal-mode-cga-template.md`。

## 9. 仓库事实消费规则

本手册不维护模块清单、技术栈、API 路径、workflow 配置面、测试矩阵或架构护栏的副本。CGA 和验收必须按本轮影响范围读取对应稳定事实源，并只在本轮产物中引用必要结论：

- 模块职责、开发命令和 New Agents 高优先级架构原则：`AGENTS.md`。
- 当前架构、服务拓扑、运行链路和模块细节：`docs/ARCHITECTURE.md`。
- HTTP/SSE/API 契约和请求响应结构：`docs/api-contracts.md`。
- 测试分层、New Agents 覆盖要求、真实模型 smoke 和 E2E / LLM judge 规则：`docs/TESTING.md`。
- 编码、设计、安全和失败显式化要求：`docs/CODING_STANDARDS.md` 与 `docs/DESIGN_PRINCIPLES.md`。

如果这些稳定事实源已经说明某条规则，本手册和本轮 spec / plan 只引用路径和本轮适用结论，不复制原文。发现事实源之间重复或冲突时，优先按第 4 节裁决，并把“是否需要收敛重复文档”作为后续文档候选记录。

## 10. 验证与记录

TDD 的具体执行方式由 `AGENTS.md` 和 Superpowers `test-driven-development` 技能承载，本手册不复述。目标模式只补充三条调度规则：

- 代码或行为变更必须在 plan 中说明将如何触发对应 TDD / 验收流程。
- 纯文档工作可以用文档核对清单替代代码测试，但必须说明没有运行代码测试的原因。
- 最终提交前运行仓库全量本地自动化。目标模式默认使用 `./scripts/test/test-local.sh all`；该脚本的覆盖范围和 CI 等价关系由 `docs/strategy/goal-mode-ci-verification.md` 与 `docs/TESTING.md` 维护，本手册不重复列举。
- LLM judge 是质量门禁时，默认最低通过分数为 80。低于 80 分必须按 judge 反馈逐项做差距分析，优先修复真实产物质量、契约、prompt 或证据缺失，再重跑 judge；不得仅关闭 judge 后声称质量门禁通过。若本轮只运行关闭 judge 的确定性回归，必须明确它只证明确定性路径，不证明 LLM 质量门禁。任何被引用过的 judge 分数都必须进入当前轮记录，不能只停留在对话上下文或命令输出里。
- 子智能体调度也是验证记录的一部分。每轮必须说明是否存在可并行只读审查、验证或互不重叠实现切片；存在且环境支持时必须实际尝试分发，不存在、不适合或环境失败时记录原因。LLM judge 低分、跨 workflow 覆盖矩阵、契约同步、浏览器级证据审查和大 dirty diff ownership 审查属于默认优先考虑子智能体的场景。

文档核对清单至少检查：

- 文档路径、链接和引用存在。
- 文档没有待办标记、未决标记或未解释的占位。
- 文档不与 `AGENTS.md`、`docs/index.md`、`docs/TESTING.md` 或当前代码事实冲突。
- 如果文档定义后续执行规则，必须说明它如何被目标模式读取和执行。

常用验证命令和 CI 等价验证规则见 `docs/strategy/goal-mode-ci-verification.md`。禁止模式、编码约束、安全要求和提交边界以 `AGENTS.md`、`docs/CODING_STANDARDS.md` 和 `docs/DESIGN_PRINCIPLES.md` 为准。

目标模式产物默认使用中文。英文只保留命令、路径、API、schema、类型名、包名、协议名和不可翻译专有名词。

建议产物位置：

- 本轮设计、计划和验证记录：优先压缩写入相关 `docs/todos/*.md` 的本轮记录。
- 长期路线图、专项计划或稳定执行规则：`docs/plans/*.md`。
- 通用目标模式规则：`docs/strategy/*.md`。

如果本轮改动改变了用户主路径、部署方式、API 契约、测试门禁或 Agent 工作流，必须判断是否同步 `docs/index.md`、`docs/ARCHITECTURE.md`、`docs/api-contracts.md`、`docs/TESTING.md`、`docs/component-inventory.md`、`docs/deployment-guide.md`、`docs/todos/` 或相关 `docs/plans/*.md`。如果决定不更新，收尾说明中要写原因。

## 11. 提交与 PR

普通任务默认不要自动 commit，除非用户本轮明确要求或任务本身规定必须提交。目标模式不同：每个完成必要实现、验证和文档记录的完整用户故事、独立工程信任闭环或可独立说明价值的切片，都应在进入下一切片前先完成批量验证、创建聚焦 commit，并在 GitHub 远端可用且没有明确暂缓条件时及时 push。这个提交节奏是目标模式的默认交付边界，不应把多个用户可感知能力包或上万行 diff 长期堆积在本地。

提交边界：

1. 先确认 `git status -sb`，只 stage 本轮相关文件。
2. 先运行本轮必要聚焦验证、仓库全量本地自动化和 CI 等价映射，再提交；如果全量自动化失败，默认不得 commit 或 push，除非本轮任务就是记录/隔离该失败并已在收尾说明中明确风险。
3. 不把无关脏文件、生成包、临时缓存或用户未确认的删除一起提交。
4. 一个 commit 应对应一个用户可感知能力闭环、一个独立工程信任闭环或一个 bugfix。
5. 子智能体 / worker 不自行 commit 或 push；主 Agent 审查、集成并验证后按用户故事形成 commit。
6. 每个稳定切片 commit 后默认应及时 push 到 GitHub；不得把多个切片合并成一个隐藏批次统一推送。若当前切片过大到需要多个 checkpoint commit，必须先重新切分为多个同级切片。
7. 如果 push 失败或暂缓，必须写明是用户要求、验证未完成、远端权限 / 凭证、网络、分支策略、无关脏改动还是其他原因；不得只说“未推送”。
8. 收尾说明必须区分“最近 commit 的统计”和“当前未提交 diff”，并报告最新 commit SHA、`HEAD` 是否等于 `origin/<branch>`、当前未提交文件列表。

如果单个 commit 预计超过约 800 行、触达 8 个以上源文件、同时跨前端和后端运行时，或包含代码加测试加文档三类改动，必须先重新评估当前切片是否过大。若过大，应按自然业务边界拆成多个同级切片；只有当拆分会破坏可编译状态、割裂同一原子契约变更，或显著增加回归风险时，才允许保持一个 commit，并在收尾说明中解释原因。

PR 描述要求以 `AGENTS.md` 为准；目标模式收尾说明只补充用户故事边界、验证证据、CI 等价覆盖和未提交 / 未推送状态。

## 12. 目标模式提示词模板

```text
请按照 `AGENTS.md` 和 `docs/strategy/goal-mode-playbook.md` 进入目标模式，持续推进 AI4SE。

本轮目标：
- <写明长期目标或当前阶段目标>
- 如果没有更高优先级用户指令，优先从 `docs/todos/`、当前失败证据、活跃计划和相关 backlog 中选择最高优先级、尚未完成、当前可推进的工作项。
- 即使已有主要目标，也要扫描 `docs/todos/` 的 P0/P1 条目；完整 CGA 时纳入候选，目标承接检查时作为改道条件记录。

执行要求：
- 先读取 `AGENTS.md`、`docs/strategy/goal-mode-playbook.md`、`docs/index.md` 和当前目标相关深层文档。
- 检查 `git status --short`，保护用户已有改动。
- 第一个实质性产物必须是完整 Current State Gap Analysis；若本轮是已确认目标轮次的连续执行，则必须输出目标承接检查。
- CGA 或目标承接检查必须记录未关闭质量门、用户最新反馈和子智能体 / 旁路审查决策；若存在 LLM judge 低于 80 分、阻断测试失败或用户明确指出的遗漏，必须先修复或记录环境阻塞，不得继续后续轮次。运行中收到用户纠错或质疑时，先中断当前推进并重新做改道判断。
- 每轮只选择或承接一个可验收用户故事；该用户故事必须是用户可感知的完整能力包，或有独立价值的工程信任闭环。
- 选定或承接用户故事后，先做中文自问自答头脑风暴，再生成中文 spec。
- 代码或行为变更按 `AGENTS.md` 和 Superpowers `test-driven-development` 技能执行；纯文档工作用核对清单替代代码测试并说明原因。
- 运行相关验证；无法运行的验证要说明原因。
- 完成型代码用户故事在 commit / push 前默认运行 `./scripts/test/test-local.sh all`；纯文档、小范围无代码、环境阻塞或我明确要求跳过时才可例外，并说明风险。
- commit / push 前运行本轮改动对应的本地 CI 等价验证；未运行或失败必须说明，并默认暂缓 push。
- commit / push 前做提交归属检查：列出当前大 dirty diff 的归属摘要和 `git diff --cached --name-only`，确认只 stage 本轮文件；无关脏改动本身不能阻止本轮聚焦提交。
- 若存在可并行读取、审查、验证或互不重叠实现切片，且子智能体工具可用，必须实际尝试分发；分发失败时重试一次更小任务，仍失败则记录环境降级并由主 Agent 顺序完成。
- 除非我明确要求不要 commit / push，否则完成型用户故事、独立工程信任闭环或可独立说明价值的切片应形成聚焦 commit，并在 GitHub 远端可用时及时 push。

请开始执行，并在关键节点给出简短进度说明。
```
