# AI4SE 目标模式运行手册

本文是 Codex / AI Coding Agent 在目标模式下持续演进 AI4SE 的运行手册。它用于减少目标提示词长度，把稳定执行方式沉淀为可版本化文档，并避免每轮都重新解释仓库结构、事实源、选题、验证和收尾规则。

## 1. 定位

AI4SE 是一个 Docker Compose 驱动的模块化 monorepo，包含统一门户、意图测试工具、New Agents 智能体工作台、共享配置和数据库工具。目标模式的职责不是机械执行任务清单，而是基于当前代码、文档、测试和用户最新目标，持续选择一个最有价值、可验收、风险可控的纵向切片推进。

目标模式由以下文件共同驱动：

- 启动提示词：只负责启动目标模式、指定本轮长期 / 短期目标入口，并要求第一步产出 Current State Gap Analysis。
- `AGENTS.md`：仓库级硬约束。本手册不重复其中的编码、架构、测试和提交细则。
- `docs/index.md`：项目知识库入口，说明 AI4SE 的模块、架构、API、数据、部署和深层文档导航。
- `docs/strategy/goal-mode-playbook.md`：目标模式通用运行规则，也就是本文档。
- `docs/todos/`：长期演进 todo，记录跨目标模式持续消化的高优先级产品、工程、测试和评判事项。
- `docs/plans/`：专项计划和专项目标模式规则。只有当本轮目标指向对应主题时，才把相关计划作为事实源展开。
- `docs/superpowers/specs/` 与 `docs/superpowers/plans/`：近期目标模式 spec / plan 产物。

提示词应保持短小，只引用稳定文件，不重复本手册、`AGENTS.md` 或深层需求文档中的细则。具体读取顺序、选题规则、产物位置、验证命令、提交边界和停机条件都由本手册承载。

## 2. 目标与授权边界

长期目标是让 AI4SE 保持为一个可本地部署、可演示、可持续演进的 AI for Software Engineering 工具集：

- `tools/frontend/` 提供统一入口和产品门户。
- `tools/intent-tester/` 提供 Flask + MidScene / Playwright 的意图测试能力。
- `tools/new-agents/` 提供结构化 Agent Runtime、typed SSE、多阶段工作流和可信 artifact 输出。
- `tools/shared/`、`scripts/`、`docs/` 提供共享配置、运维、测试和知识库基础设施。

如果用户没有指定本轮目标，默认从 `docs/todos/`、当前失败证据、活跃计划和相关 backlog 中选择最高优先级、尚未完成、当前上下文可推进的工作项；但每轮仍必须先做 Current State Gap Analysis，不能直接套用上一轮结论或固定清单。

如果用户指定了本轮主要目标，也必须快速扫描 `docs/todos/` 中的 P0/P1 条目。若存在与当前目标同向、低冲突、可形成纵向切片的高优先级 todo，应在 CGA 中纳入候选；若本轮不消化，必须说明暂缓原因和后续去向。

技术债、New Agents 专项演进、部署治理、评判体系等都属于可被目标模式处理的一类需求；只有当用户目标、todo 或 CGA 候选指向这些主题时，才读取对应专项计划，例如 `docs/plans/tech-debt.md` 或 `docs/plans/goal-mode-tech-debt-rules.md`。

目标模式可以在符合 `AGENTS.md`、本手册、相关深层文档和当前代码事实的前提下，自主完成上下文读取、gap 排序、spec、plan、TDD 实现、验证和记录更新。

不要把用户确认作为常规节点。普通澄清由 Agent 基于文档、代码、测试和合理 assumptions 自行回答，并在 spec 或收尾说明中记录。

只有以下情况才停止并向用户汇报：

- 需要用户提供权限、凭证、外部服务访问、浏览器登录态或仓库访问。
- 当前任务需要联网安装依赖、访问模型供应商或调用外部 API，而环境未授权。
- 连续阻塞，且无法通过当前文档、代码、测试或合理 assumptions 推进。
- 用户最新消息明确要求暂停、只讨论、不改代码或等待确认。
- 当前架构、文档事实或用户目标之间存在无法自行裁决的根本冲突。

## 3. 启动协议

每轮目标模式启动后，先执行启动协议：

1. 读取 `AGENTS.md`。
2. 读取本手册。
3. 读取 `docs/index.md`，获取当前项目结构和深层文档导航。
4. 检查 `git status --short`。如果存在未提交变更，必须判断它们是用户改动、上一轮遗留还是当前轮输入；不得覆盖、回滚或格式化不属于本轮的变更。
5. 按任务方向读取相关事实源：架构、API、测试、编码规范、设计原则、todo、专项计划、已有 spec / plan、当前代码和测试。
6. 第一个实质性工作产物必须是 Current State Gap Analysis。

不要先写 implementation plan、先改代码、先跑固定 todo 清单，或直接沿用上一轮记忆。

### 3.1 工作树隔离原则

目标模式中的每一个可写工作都必须优先在独立 git worktree 中执行，避免多个 Agent、子智能体、自动续跑或人工操作在同一个工作区互相覆盖代码、暂存区和构建产物。

启动协议完成 `git status --short` 后，必须执行 worktree 隔离检查：

1. 检测当前目录是否已经是独立 worktree；如果已经隔离，记录当前路径和分支，并继续本轮工作。
2. 如果当前目录不是独立 worktree，优先使用平台原生 worktree / 分支隔离能力；没有原生能力时，再使用 `git worktree` 创建 project-local worktree。
3. project-local worktree 目录必须位于已被 git 忽略的目录中，例如 `.worktrees/`；如果未忽略，先补充忽略规则，再继续创建。
4. 主 Agent、可写 Worker 子智能体和需要修改文件的验证任务，都必须在各自独立 worktree 中工作；只读 Explorer 可以在当前目录读取，但不得修改文件。
5. 如果当前会话已经在非隔离工作区产生了活跃改动、用户明确要求沿用当前工作区，或沙箱/权限阻止创建 worktree，可以降级为当前工作区继续执行；但必须在 CGA 和收尾说明中写明降级原因、当前 `git status --short`、以及如何避免覆盖他人改动。
6. 不得因为任务简单、只改一两个文件或已经理解问题就跳过隔离检查。隔离是目标模式默认前置条件，不是大型任务才需要的优化。

## 4. 事实源与冲突优先级

每轮判断都必须基于当前文件，而不是上一轮记忆。

事实源默认顺序：

1. `AGENTS.md`
2. `docs/strategy/goal-mode-playbook.md`
3. `docs/index.md`
4. `docs/ARCHITECTURE.md`
5. `docs/api-contracts.md`
6. `docs/TESTING.md`
7. `docs/CODING_STANDARDS.md`
8. `docs/DESIGN_PRINCIPLES.md`
9. `docs/integration-architecture.md`
10. `docs/component-inventory.md`
11. `docs/todos/`
12. 与本轮相关的 `docs/plans/`、`docs/superpowers/specs/`、`docs/superpowers/plans/`、测试需求、当前代码、测试、脚本和 git 历史。

如果某一轮只涉及特定模块，可以按需缩小读取范围，但必须在 CGA 中说明实际读取了哪些关键文件，以及哪些事实源因无关而未展开。

冲突裁决：

- 用户最新明确要求优先于历史文档。
- 当前代码和当前测试优先于过期文档。
- `AGENTS.md` 和本手册优先于普通计划文档。
- `docs/TESTING.md`、`docs/CODING_STANDARDS.md`、`docs/DESIGN_PRINCIPLES.md` 是工程护栏，不应被局部实现便利覆盖。
- 专项计划和专项规则只对其声明的目标范围生效；它们不能替代本手册的通用启动协议，但可以补充对应主题的执行规则。

发现文档与代码冲突时，不要默默选择一边。必须在 spec、plan 或收尾说明中写清楚冲突、取舍和后续处理。

## 5. 每轮循环

每轮按以下顺序执行：

1. 重新读取事实源和相关代码，不依赖上一轮记忆。
2. 检查 git 状态并保护用户已有改动。
3. 执行 Current State Gap Analysis：列出候选 gap、排序依据、未选候选去向、边界和验收证据。
4. 选择一个且只选择一个 milestone。milestone 必须是用户可感知的完整能力包，或能独立降低后续演进风险的工程信任闭环；不得把单个控件、单个字段、单个 helper、单个 parser 分支或单条测试当作独立 milestone，除非它本身解除阻断、风险或证据失真。完成后必须能用一句“用户现在可以……”描述新增能力或可信闭环。若候选无法通过切片厚度门禁，不得继续写 spec / plan，必须回到 CGA 聚合相邻缺口或暂缓。
5. 写中文 spec，包含用户故事、场景、验收条件、风险、文件范围和验证计划。
6. 写中文 implementation plan，拆成 TDD 可执行步骤。
7. 先写失败测试或等价验收检查，再实现最小改动，再重构清理。
8. 运行与本轮改动匹配的最小验证；触及共享行为、跨层协议或用户主路径时扩大验证范围。
9. 更新必要的文档记录，例如 `docs/todos/`、相关专项计划、spec / plan 或测试策略说明。
10. 目标模式下，一个通过验证的完整 milestone 默认必须形成聚焦 commit；如果用户明确要求不要提交、当前只是临时探索、验证未完成或存在未裁决风险，才可以暂不提交，并必须在收尾说明中写明原因。普通非目标模式任务仍遵循“未明确要求不要自动 commit”的默认边界。
11. 收尾说明必须列出改动、验证命令、未运行验证及原因、残余风险和下一步候选。

### 5.1 子智能体加速规则

目标模式鼓励使用子智能体加速开发，但子智能体是并行执行工具，不是跳过 TDD、系统性调试或验证门禁的理由。主 Agent 始终负责选题、边界、最终集成和完成声明。

优先使用子智能体的场景：

- **Explorer 子智能体**: 用于并行读取独立事实源、定位候选缺口、审计文档冲突、梳理测试失败分组。Explorer 默认只读，不改文件；输出必须包含证据路径和结论边界。
- **Worker 子智能体**: 用于已有明确 spec / plan、且写入范围彼此独立的实现任务。分发时必须说明文件所有权、测试命令、禁止回滚他人改动，并要求直接在其工作区完成补丁。
- **Reviewer / Verification 子智能体**: 用于实现后做 spec 符合性审查、代码质量审查、风险点复核或并行运行独立验证。审查结果不能替代主 Agent 的最终 diff 检查和必要验证。

默认流程：

满足下列默认触发点时应分发子智能体；如果因为任务过小、执行环境不支持、上下文不足或用户只读要求而跳过，必须在 CGA 或收尾说明中记录原因。

1. 在 CGA 阶段，如果存在 2 个以上独立候选或多个事实源可并行读取，分发 explorer 子智能体并行审计。
2. 在 implementation plan 阶段，如果任务可拆成互不重叠的文件或模块切片，优先采用 `superpowers:subagent-driven-development`：每个任务由新 worker 实现，再经过 spec review 和 quality review。AI4SE 仓库规则覆盖该技能示例中的自动 commit 行为；worker 不得创建 commit，除非用户本轮明确要求提交。
3. 在多处失败彼此独立时，采用 `superpowers:dispatching-parallel-agents`：按失败文件、子系统或问题域分发，不让多个子智能体同时修改同一文件集合。
4. 子智能体运行期间，主 Agent 继续做不重叠工作，例如补文档、准备验证、审查其他模块；不要空等。主 Agent 不得同时修改 worker 所有权范围或共享记录文件；如必须更新 `docs/todos/`、专项计划、当前 plan 或 playbook，先把这些文件纳入主 Agent 保留范围，待 worker 返回后统一集成。
5. 子智能体返回后，主 Agent 必须审查其变更、解决冲突、运行必要验证，并把实际结果写入收尾说明。

如果用户要求“持续消化 backlog”但没有指定下一个具体工作项，主 Agent 应优先把只读 explorer 用作候选发现器：每个 explorer 聚焦一个独立问题域，返回可复现现象、证据路径、建议 RED 用例或验收检查、风险边界。主 Agent 只从这些候选中选择一个当前切片进入实现；未选候选必须写入来源 backlog、当前计划或下一轮候选说明，避免子智能体发现被丢失。

当前仓库经常处于大规模未提交变更状态。分发任何可写 worker 前，主 Agent 必须先记录 `git status --short`，在提示中列出允许写入路径，并明确禁止 worker 修改该范围之外的文件。worker 返回后，主 Agent 必须用 `git diff -- <允许路径>` 和 `git status --short` 核对实际写入范围；如果发现越界改动，先隔离并审查，不得直接合并到当前切片。

### 5.2 Worktree 化子智能体集成

当用户明确鼓励并行开发，或目标模式需要同时消化多个独立切片时，默认采用“一个可写子智能体一个 git worktree”的方式，而不是让多个 worker 在同一个工作区直接修改文件。

推荐调度方式：

1. 主 Agent 保留当前主工作区作为 dispatcher / integrator，只负责拆任务、维护共享 todo / playbook / plan、审查结果、合并代码、运行最终验证和提交。
2. 每个可写 worker 在独立 worktree / forked workspace 中工作，分支名使用 `codex/<主题>` 或用户指定前缀。worker 不得直接在主工作区修改文件，不得提交或推送，除非用户本轮明确要求它这样做。
3. 每个 worker 的任务提示必须包含：
   - 业务目标和验收条件。
   - 允许写入路径和禁止写入路径。
   - 需要先写失败测试、再实现、再运行的验证命令。
   - “你不是唯一改代码的 agent；不要回滚或覆盖他人改动；如发现范围外问题，只报告不要修”的约束。
   - 完成后必须列出改动文件、测试结果、建议提交边界、风险和未完成项。
4. 多个 worker 可以并行启动，但写入范围必须尽量互不重叠。高冲突文件，例如 `ArtifactPane.tsx`、大型 store、共享 todo 和 playbook，默认由主 Agent 最后统一接线；worker 优先负责纯函数、核心模块、服务层、测试文件或独立页面。
5. worker 返回后，主 Agent 按顺序集成：
   - 先读最终说明，确认没有 `NEEDS_CONTEXT`、`BLOCKED` 或未运行验证。
   - 检查 worker worktree 中的实际 diff 和写入范围。
   - 以 cherry-pick、patch 或手工移植方式合入主工作区。
   - 每合入一个 worker 结果，运行该切片最小验证。
   - 全部合入后运行相关测试、lint、build 和 `git diff --check`。
6. worktree 只能降低并发编辑污染，不能消除最终语义冲突。若两个 worker 触达同一设计面或同一用户流程，主 Agent 必须在集成阶段重新审查行为一致性，不得仅凭 git merge 成功判定完成。
7. 如果环境不支持 worktree 或当前主工作区已有必须沿用的未提交切片，主 Agent 可以临时降级为串行主线程实现；但必须在收尾说明中记录降级原因，并把 worktree 化并行留到下一切片。

子智能体验收至少包含三步：先读其最终说明，确认是否有 `NEEDS_CONTEXT`、`BLOCKED` 或未运行验证；再检查实际 diff 是否满足 spec 且没有额外重构；最后由主 Agent 在当前上下文重新运行本切片最小验证。只有这三步都完成后，子智能体结果才能进入完成记录。

不要使用子智能体的场景：

- 问题尚未分清边界，多个症状可能来自同一个根因。
- 任务需要连续修改同一文件或共享状态，多个 worker 会互相覆盖。
- 只是单行修复、单个聚焦测试或主 Agent 已经持有完整上下文，分发成本高于收益。
- 子智能体需要凭证、联网、浏览器登录态或外部权限，而主 Agent 尚未取得用户授权。
- 当前执行环境没有可用子智能体或任务分发能力；此时不要强行模拟并行，改为顺序执行只读探索或实现，并记录降级原因。

子智能体输出必须被当作工程输入，而不是完成证明。任何“已修复”“测试通过”“可合并”的说法，都必须由主 Agent 在当前上下文重新核对证据后才能对用户声明。

## 6. Current State Gap Analysis

CGA 的目标是基于最新代码、测试、文档和用户目标，判断下一步最有价值、最可验收、最少破坏当前契约的工作包。

每轮 CGA 必须包含：

- 事实源快照：本轮实际读取了哪些关键文件。
- 能力包聚合：先把 todo 子项、用户反馈、测试失败和技术缺口归并到同一用户场景或工程信任闭环下，再形成候选；不得从单个子字段、单按钮、单 endpoint 或单测试直接进入 milestone 选择。CGA 的最小候选单位必须是“完整用户功能厚切片”：用户从一个真实入口开始，经过必要动作、系统处理、可见反馈、状态承接和失败恢复后，能完成一段有独立价值的功能闭环。
- 候选 gap：至少 2 个候选能力包；如果确实只有 1 个，也要说明为什么其他缺口无法在本轮形成能力包。
- 排序依据：为什么选中的 gap 比其他候选更适合作为本轮 milestone。
- 用户功能包边界：明确本轮代表的完整用户功能是什么，包含哪些同一用户意图下的相邻缺口；如果排除相邻缺口，必须解释为什么它属于不同用户意图、不同风险面或不同验证闭环。不能因为实现方便，把同一用户动作链中的后端契约、前端入口、状态保存、错误提示、下游消费或验证证据拆成连续多轮目标模式。
- 切片准入判断：说明选中 milestone 是否值得消耗完整 Superpowers 流程成本，是否形成用户可感知的完整动作链，是否能一次性消化同一用户场景下的相邻小缺口，以及完成后用户可观察能力相对当前状态的实际增量。若一个候选只能带来“局部更好”而不能让用户多完成一段真实任务，默认必须继续向外聚合，直到形成完整用户功能包，或改选其他候选。
- 切片厚度门禁：明确本轮是否通过“入口、动作、系统处理、可见结果、状态承接、失败反馈、证据”检查；如果缺失其中关键环节，必须说明如何扩大边界，或为什么这是受控工程信任闭环例外。
- 未选候选去向：保留为下一轮候选、写入来源 backlog / 专项计划、归入更大工作包、标记 blocked / wont-do，或明确暂不处理。
- 验收口径：每个候选都要说明可以被什么证据验证。
- 边界判断：哪些内容进入本轮，哪些内容不进入本轮。

## 7. Milestone 选择与粒度

AI4SE 不维护固定路线图。目标模式选题时，应同时考虑用户最新目标、当前代码事实、todo / backlog 优先级、测试可行性和用户可观察价值。

Superpowers 流程成本较高，因此 milestone 粒度默认应偏厚。一次目标模式切片应尽量覆盖同一用户场景下连续相邻的小缺口，形成一个用户可感知的能力包；不要为了快速推进进度，把同一能力链拆成多个“按钮级”“字段级”“helper 级”切片。默认情况下，一个 milestone 至少应能让用户完成一段新的真实操作，或让调用方获得一个可复用、可验证、会改变后续决策的能力。

目标模式的默认切片单位是“完整用户功能厚切片”，不是“代码改动集合”。完整用户功能厚切片必须围绕一个真实用户目标命名，例如“生产环境可视化稳定呈现闭环”“模型供应商异常自检与修复闭环”“Artifact 编辑与生成并发承接闭环”。它应同时包含这个目标下必要的入口、操作、系统处理、可见结果、状态承接、失败恢复和证据。实现时可以按多个 RED / GREEN 小步推进，但 CGA、spec、plan、验收和 commit 边界必须围绕完整功能包组织。

厚切片的默认目标不是“改动足够多”，而是“完成后用户或调用方获得一段完整可用的新能力”。如果某个候选只能让整体进度增加 0.x% 或只能被描述为内部准备工作，CGA 必须先尝试扩大边界，把同一动作链中的后端契约、服务调用、前端状态、可见反馈、错误处理、测试证据和必要文档合并进同一 milestone。只有当扩大后会跨越不相关用户场景、显著增加破坏面，或当前候选本身解除阻断 / 风险 / 证据失真时，才允许保持较小边界。不能为了降低单轮工作量而把可连续交付的同一用户场景拆成多个目标模式轮次。

如果候选 milestone 只能用内部算法、单个渲染分支、单个三方 merge 场景、单个错误分类或单个结构化字段来命名，默认判定为过薄。CGA 必须把它上升到用户功能包命名，例如从“同章节段落插入与改写自动合并”上升为“Artifact 冲突处理收尾能力包”，并把安全自动处理、无法自动处理时的解释、用户手工承接、状态恢复和验证证据一起纳入边界。实现内部可以按多个小 RED / GREEN 步骤递进，但 CGA、spec、plan、验收和 commit 边界应围绕完整用户功能包组织。

如果一个候选看起来只剩“最后一点”，仍不能默认作为薄切片进入实现。CGA 必须先问：这“一点”是否只是某个用户功能闭环的缺失环节？如果是，必须把闭环中的相邻缺口一起纳入；如果不是，必须说明它解除的阻断或证据风险。不要用“剩余工作不多”“顺手补一下”“推进百分比”作为切片成立理由。

同一用户意图下的相邻缺口不得默认留给下一轮。例如用户要“编辑后的 Artifact 能可靠承接后续生成”，自动合并一个安全场景、阻止一个不安全场景、解释为什么无法合并、提供用户可操作的承接路径、恢复后状态一致，属于同一功能包的不同必要环节。只有当相邻缺口需要不同入口、不同角色、不同数据契约、不同服务依赖，或会显著扩大回归面时，才允许拆出，并且必须在 CGA 中写出拆分理由和后续去向。

milestone 选择必须按“能力包 -> 用户动作链 -> 验收证据 -> 实现边界”的顺序收敛，而不是按“还有哪些小 todo -> 本轮挑一个小 todo”的顺序推进。todo 可以保留细粒度缺口，方便持续追踪；CGA 不能把这些细粒度缺口原样当作本轮 milestone。若一个 todo 子项必须依附另一个子项用户才看得见价值，应在同一切片中合并；若合并后工作量过大，应改选另一个可独立交付的能力包，而不是把半条链路先做掉。

### 7.1 切片厚度门禁

CGA 选择 milestone 前必须执行切片厚度门禁。门禁不是按改动文件数量判断，而是按完成后是否形成真实能力判断。默认应先寻找能够一次性交付完整用户功能的厚切片；只有当用户功能包过大且可以按自然业务边界拆分时，才允许拆成多个 milestone。自然业务边界必须表现为不同用户目标、不同入口、不同下游承接或不同风险面，而不是不同文件、不同组件、不同函数或不同测试。

默认通过标准：

- 有明确用户入口或调用方入口。
- 有清晰的用户目标和完成状态，而不是只说明要改哪个模块。
- 有一个用户或调用方会主动触发的真实动作，而不是单纯内部预备工作。
- 系统完成可描述的核心处理，并产生可观察结果。
- 结果能被保存、恢复、下游消费、评判或作为后续证据承接。
- 失败路径能显式停止并说明原因，不靠 silent fallback。
- 验证证据覆盖入口、处理、结果和关键失败路径。
- 完成后能用一句“用户现在可以……”或“调用方现在可以……”说清楚新增能力。
- 同一用户动作链中没有明显被推迟到下一轮的必要环节；如果有，必须解释技术阻断或风险边界。

默认不通过的信号：

- milestone 名称只能写成“新增字段”“补一个 endpoint”“修一个 parser”“调整一个按钮”“补一条测试”。
- milestone 名称只能写成某个内部算法分支或局部异常场景，例如“某类段落 merge”“某种 Mermaid 语法变体”“某个 provider 错误码”。
- milestone 名称只能写成某个页面局部、状态块、菜单项、提示词段落、导出格式、缓存策略或错误码。
- 验收条件只验证内部函数，不验证 API、页面、工作流、下游消费或可见反馈。
- 完成后只能汇报百分比增加 0.x%，但说不清用户多完成了什么。
- 当前切片显然只是某个用户动作链的前半段，而后半段没有技术阻断却被留到下一轮。
- 需要连续多轮目标模式才能让用户第一次看到或使用同一能力。
- 同一用户功能需要后端、前端、状态、错误反馈和验证证据联动，但 CGA 只选择其中一层。

不通过时的处理顺序：

1. 先扩大边界，把同一用户场景下相邻的后端契约、前端服务、页面交互、状态持久化、错误反馈、下游消费和验证证据并入同一 milestone。
2. 再检查扩大后的 milestone 是否仍是同一个用户目标。只要入口、动作、状态承接和失败恢复仍服务于同一目标，默认保持合并，不拆薄。
3. 如果扩大后会跨越不相关用户场景或显著增加破坏面，把该候选记录为更大能力包的子项，并改选更合适的 milestone。
4. 如果它是工程信任闭环例外，必须写清楚解除的阻断、调用方受益、后续能力如何依赖它，以及为什么不能与用户可见能力同轮交付。
5. 如果仍无法形成真实能力包，暂缓，不进入 spec / plan。

优先级默认如下：

1. 用户最新明确目标。
2. 阻断主路径演示、使用或部署的 P0 问题。
3. 会导致 silent fallback、协议混用、密钥泄露、错误写入或证据失真的工程风险。
4. `docs/todos/` 中 P0 且能提升系统专业可信度、能力评判或后续目标模式选题质量的工作项。
5. 当前活跃专项计划或 backlog 中高优先级且可形成纵向切片的工作项。
6. 能提高后续目标模式迭代可信度的测试、验证和文档护栏。
7. 普通清理、性能优化和历史文档整理。

一个合格 milestone 应能回答：

- 这个 milestone 对应的完整用户功能叫什么，用户为什么会主动使用它。
- 用户或调用方从哪里开始。
- 用户看到什么状态、说明、选择或风险。
- 用户做出什么决定，或调用方完成什么动作。
- 系统执行什么核心能力。
- 成功后接到主流程哪里。
- 失败时如何显式停止、解释原因并给出下一步。
- 有哪些 evidence 可以证明这条链路真的走通。
- 完成后能否用一句“用户现在可以……”说清楚新增能力。

一个合格 milestone 还应满足至少一项：

- 交付一个用户可直接使用的新功能或完整体验改进。
- 打通一条原本断裂的主路径或跨模块链路。
- 把一个只读能力升级为可操作、可保存、可恢复或可评判的闭环。
- 把一组分散的底层能力集成为调用方可复用的 API / 页面 / 工作流能力。
- 消除会造成 silent fallback、错误写入、安全风险、证据失真或无法验证的工程阻断。

如果 CGA 发现候选 milestone 只会交付“一个小控件、一个小字段、一个样式点、一个 parser 分支、一条服务函数或一条测试”，默认不能进入 spec / plan。此时必须先尝试把它合并进最近的用户动作链；如果无法合并，就暂缓并记录为更大能力包的子项。

CGA 还必须避免把一个完整能力链拆成多个连续目标模式轮次来“刷进度”。下列内容通常应在同一切片内合并，而不是拆成独立 milestone：

- 一个资源从只读到可操作：API 写入、前端服务、页面交互、保存反馈、失败提示和刷新后可恢复。
- 一个工作流从生成到消费：artifact 生成、结构化解析、用户可见入口、下游导入 / 执行 / 评判和证据记录。
- 一个质量规则从提示词到可验证：prompt / contract、失败测试、成功样例、前端展示或 judge 输入，以及收尾证据。
- 一个列表从展示到可管理：筛选 / 搜索、选择模型、关键编辑动作、批量反馈、错误状态和持久化验证。
- 一个配置问题从发现到修复：运行时异常归因、可见诊断、设置入口、连接检测、保存 / 轮换、重试承接和观测证据。
- 一个可视化问题从生成到稳定呈现：生成契约、结构化校验、安全渲染、超时 / fallback、用户可读错误、重试路径和生产复现证据。
- 一个 Artifact 协作问题从冲突到继续推进：冲突检测、可读差异、安全自动处理、危险场景拦截、手工承接、保存 / 恢复和后续阶段消费。

常见过薄切片及处理方式：

| 过薄切片 | 为什么过薄 | 应并入的厚切片 |
| --- | --- | --- |
| 只新增 schema / 类型 | 只有机器契约，没有用户可见结果和消费链路 | 并入 API / SSE / 前端状态写入 / 测试证据闭环 |
| 只修一个 parser 分支 | 用户只获得错误分类，主流程没有新增可理解能力 | 并入请求边界、错误文案、端点测试和 UI 可见错误 |
| 只改 prompt | LLM 输入改变但用户价值不可证明 | 并入 artifact 质量、契约校验、真实或 mock 输出验收 |
| 只补测试 | 验证加强但能力不变 | 并入对应用户故事的 TDD / 验收任务；工程信任闭环例外需说明独立价值 |
| 只更新文档措辞 | 不改变后续执行行为或当前能力边界 | 并入当前主路径文案、playbook 规则或稳定事实源校准 |
| 只新增一个按钮或入口 | 用户只能到达半成品页面或触发单点动作，没有完整处理链 | 并入页面加载、状态展示、核心操作、错误处理和返回路径 |
| 只给已有列表增加一个批量字段操作 | 用户感知价值过窄，完整流程成本不划算 | 并入资产中心、选择模型、批量操作、结果反馈和持久化验证 |
| 只补一个 Artifact 三方 merge 场景 | 用户真正需要的是编辑与生成并发时的可靠承接，不是某个算法分支 | 并入 Artifact 冲突处理能力包：安全自动合并、危险场景拦截、可读原因、手工承接、保存 / 恢复和验证证据 |
| 只补一个 Mermaid 语法兼容分支 | 用户真正需要的是生产环境可视化稳定显示与失败可恢复，不是单条语法规则 | 并入可视化稳定呈现能力包：契约约束、安全渲染、缓存 / 超时、fallback、重试和生产复现证据 |
| 只补一个模型供应商错误码 | 用户真正需要的是知道为什么失败并能完成配置修复，不是看到更多内部分类 | 并入模型配置自检与修复闭环：异常归因、设置入口、连接检测、保存 / 轮换、重试和运行统计 |
| 只把整体进度推进 0.x% | 百分比增量来自拆分口径，不代表用户获得新能力 | 重新聚合剩余能力包，优先交付一条完整用户动作链或一个工程信任闭环 |

受控例外：如果某个底层缺口会导致后续工作出现 silent fallback、安全风险、证据失真或无法验证，允许作为工程信任闭环进入 milestone。但它仍必须给调用方或后续目标模式提供独立价值，不能停在内部 helper。

### 7.2 进度口径

目标模式汇报进度时，不按文件数、测试数、提交数或拆分出的微任务数估算，也不要用过细切片制造 0.x% 的进度增量。进度应基于剩余用户可感知能力包、主路径闭环和工程信任闭环的完成情况估算。

如果本轮切片本身很薄，但汇报后会把整体进度从 99.x% 推高一点点，这通常说明切片选择错误，而不是项目接近完成。百分比只能在能力包边界确定后作为粗略沟通结果，不能反向决定切片厚度。此时下一轮 CGA 必须重新聚合更厚的用户场景，或明确说明为什么剩余工作只能以阻断型工程闭环推进。

汇报百分比时必须同时说明：

- 已完成的用户可感知能力包有哪些。
- 剩余的能力包还有哪些，不把子字段、子按钮、单端点当作独立百分比单位。
- 当前选择的下一切片预计会交付哪一个完整能力包。
- 如果百分比超过 90%，仍要列出剩余能力包，避免给出“做完下一片就全部完成”的误导。

## 8. AI4SE 模块映射

不同模块的纵向切片形态不同，CGA 和验收应按模块映射。本节是 AI4SE 仓库适配参考；具体架构、API、测试和设计原则以 `AGENTS.md`、`docs/ARCHITECTURE.md`、`docs/api-contracts.md`、`docs/TESTING.md` 和 `docs/DESIGN_PRINCIPLES.md` 为准，不在本手册重复展开。

- 典型证据：组件测试、`npm run build`、截图或浏览器 walkthrough。

## 9. TDD 与实现规则

默认执行 `AGENTS.md` 中的 TDD 要求。目标模式中的落地顺序是：

1. 先写或更新失败测试，证明当前缺口存在。
2. 运行聚焦测试，确认失败原因符合预期。
3. 写最小实现让测试通过。
4. 运行聚焦测试确认通过。
5. 根据风险扩大验证范围。
6. 必要时重构，但不得改变本轮目标边界。

纯文档工作可以用文档核对清单替代代码测试，但必须说明没有运行代码测试的原因，并至少做以下检查：

- 文档路径、链接和引用存在。
- 文档没有 `TODO`、`TBD` 或未解释的占位。
- 文档不与 `AGENTS.md`、`docs/index.md`、`docs/TESTING.md` 或当前代码事实冲突。
- 如果文档定义后续执行规则，必须说明它如何被目标模式读取和执行。

禁止模式、编码约束、安全要求和提交边界以 `AGENTS.md`、`docs/CODING_STANDARDS.md` 和 `docs/DESIGN_PRINCIPLES.md` 为准。CGA、spec、plan 和收尾说明只需说明本轮如何遵守这些事实源，不要复制其完整规则文本。

## 12. 提交与 PR

普通任务默认不要自动 commit，除非用户本轮明确要求或任务本身规定必须提交。目标模式不同：每个完成 TDD、验证和文档记录的完整 milestone 都应在进入下一 milestone 前创建一个聚焦 commit，避免多个用户可感知能力包堆积成一个巨型提交。需要提交或创建 PR 时，遵循 `AGENTS.md` 的提交和 PR 规范；本手册只补充目标模式边界：

1. 先确认 `git status -sb`，只 stage 本轮相关文件。
2. 先运行本轮必要验证，再提交。
3. 不把无关脏文件、生成包、临时缓存或用户未确认的删除一起提交。
4. 一个 commit 应对应一个用户可感知能力闭环、一个独立工程信任闭环或一个 bugfix；不要把多个已完成 milestone 长期累积后一次提交。
5. 子智能体 / worktree worker 不自行 commit 或 push；主 Agent 审查、集成并验证后按 milestone commit。
6. 如果用户要求 push，则每个稳定 milestone commit 后应及时 push，或按用户指定的一组稳定 commits 统一 push；无论哪种方式，进入下一 milestone 前必须报告当前 push 状态。
7. 收尾说明必须区分“最近 commit 的统计”和“当前未提交 diff”，并报告最新 commit SHA、`HEAD` 是否等于 `origin/<branch>`、当前未提交文件列表。
8. 如果因为临时探索、验证失败、需要用户裁决或外部权限导致 milestone 暂不提交，收尾说明必须明确写出未提交原因和保留的文件清单。

提交频率门槛：

- 一个 milestone 内部如果自然拆出多个可独立验收的能力闭环、工程信任闭环或 bugfix，必须拆成多个顺序 commit；不要因为它们属于同一长期目标就合并成一个大 commit。
- 单个 commit 的 diff 如果预计超过约 800 行、触达 8 个以上源文件、同时跨前端和后端运行时，或包含代码加测试加文档三类改动，必须先重新评估是否可拆成 checkpoint commit。只有当拆分会破坏可编译状态、割裂同一原子契约变更，或显著增加回归风险时，才允许保持一个 commit，并在收尾说明中解释原因。
- 合理的 checkpoint commit 应保持仓库处于可验证状态：每个 checkpoint 至少运行与其范围匹配的最小验证，记录验证命令，并且不得依赖后续 commit 才能通过基础类型检查或单元测试。
- 文档 / playbook / todo 更新如果只是记录刚完成的代码切片，可以和对应切片同 commit；如果它改变目标模式规则、开发流程或跨切片计划，应单独 commit，方便以后追溯规则变化。
- 当用户要求 push，checkpoint commit 也应按稳定边界及时 push；长时间开发中不得把多个小时或数万行已验证改动留在本地未提交状态。

提交检查点协议：

1. 写 implementation plan 时先标注预计 commit 边界。每个边界应对应一个可验证状态，例如后端契约、前端交互、导出能力、迁移脚本、测试补强或规则文档更新。
2. 开发过程中一旦出现以下任一情况，必须暂停继续扩大 diff，先评估是否提交 checkpoint：新增或修改超过约 800 行、触达 8 个以上源文件、同时修改前后端运行时、一个子智能体结果已经完整验证、或用户明确询问“代码是否已经提交 / 为什么还没提交”。
3. 提交前必须执行 `git diff --stat` 和 `git status -sb`，并用这些数字反查 commit 粒度是否合理。若 diff 已达到数千行甚至数万行，默认判定为过大；除非能证明它是不可拆的原子生成物或单一外部制品，否则必须拆分后再提交。
4. 多个子智能体并行时，每个 worker 合入后都应形成独立候选 checkpoint。主 Agent 可以按依赖顺序合并多个 checkpoint，但不得把多个已验证 worker 结果长期积压成一个最终大提交。
5. 如果确实需要一个大 commit，收尾说明必须写清楚为什么不能拆、实际 diff 规模、验证命令和残余风险。不能只用“同属一个目标”作为合并成大 commit 的理由。
6. push 不是替代 commit 的同步手段。若用户要求提交到 GitHub，应先在本地形成聚焦 commits，再把这些 commits 推送；不要把长时间未提交的大量本地修改留到最后一次性提交。

PR 描述应包含：

- 问题背景。
- 解决方案。
- 影响模块。
- 验证命令和结果。
- UI 变化截图或说明。
- 未覆盖风险和后续工作。
