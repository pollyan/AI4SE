# 目标模式子智能体与当前工作区协作附录

本文承载目标模式中的子智能体、并行协作、当前工作区保护和集成规则。主执行入口见 `docs/strategy/goal-mode-playbook.md`。

## 1. 基本原则

子智能体是目标模式默认加速手段，不是跳过 TDD、系统性调试或验证门禁的理由。只要存在可并行读取、可并行审查或互不重叠的实现切片，主 Agent 应优先考虑分发子智能体；跳过时必须在 CGA 或收尾说明中记录原因。主 Agent 始终负责选题、边界、最终集成和完成声明。

当前 AI4SE 目标模式默认只在当前主工作区推进，不创建或切换 git worktree。主 Agent 必须记录 `git status --short`，限定本轮写入范围，只 stage 本轮相关文件。子智能体可以用于并行读取、审查和验证；可写 worker 只有在文件范围互不重叠且主 Agent 能审查 diff 时才分发。

每轮在 CGA、spec 或收尾说明中记录：

- 当前 `git status --short`。
- 本轮允许写入路径。
- 不触碰的既有脏文件。
- 如何避免覆盖已有改动。

只有用户明确要求使用独立工作区时，才允许重新讨论 worktree；否则不要创建、切换或要求 worktree。

## 2. 子智能体类型

- **Explorer 子智能体**：用于并行读取独立事实源、定位候选缺口、审计文档冲突、梳理测试失败分组或搜索 backlog 候选。Explorer 默认只读，不改文件；输出必须包含证据路径、可复现现象、建议验收检查和结论边界。
- **Worker 子智能体**：用于已有明确 spec / plan、且写入范围彼此独立的实现任务。分发时必须说明文件所有权、测试命令、禁止回滚他人改动，并要求直接在其工作区完成补丁。
- **Reviewer / Verification 子智能体**：用于实现后做 spec 符合性审查、代码质量审查、禁止模式扫描、风险点复核、浏览器级验证或并行运行独立验证。审查结果不能替代主 Agent 的最终 diff 检查和必要验证。

## 3. 默认触发点

满足下列默认触发点时应分发子智能体；如果因为任务过小、执行环境不支持、上下文不足、用户只读要求或写入范围无法隔离而跳过，必须在 CGA 或收尾说明中记录原因。

1. 在 CGA 阶段，如果存在 2 个以上独立候选或多个事实源可并行读取，分发 explorer 子智能体并行审计。
2. 在 implementation plan 阶段，如果任务可拆成互不重叠的文件或模块切片，可采用 `superpowers:subagent-driven-development`。AI4SE 仓库规则覆盖该技能示例中的自动 commit 行为；worker 不得创建 commit，除非用户本轮明确要求提交。
3. 在多处失败彼此独立时，采用 `superpowers:dispatching-parallel-agents`，按失败文件、子系统或问题域分发，不让多个子智能体同时修改同一文件集合。
4. 当 LLM judge 低于 80 分、用户指出质量门遗漏、或当前轮涉及跨 workflow 契约同步时，默认分发只读 reviewer / explorer 做差距分析、契约复核或证据审查；若环境没有子智能体或分发会扩大风险，必须写清具体原因。
5. 子智能体运行期间，主 Agent 继续做不重叠工作，例如补文档、准备验证、审查其他模块；不要空等。
6. 浏览器级验证、文档规则审查、禁止模式扫描等可以与主实现并行时，分发 verification 子智能体；最终完成声明仍以主 Agent 读取的命令输出和 diff 为准。
7. 当前工作区有大量未提交改动时，优先使用只读 explorer 和 verification 子智能体；只有可写范围能被精确限定到互不重叠文件集合时，才分发 worker。
8. 子智能体返回后，主 Agent 必须审查其变更、解决冲突、运行必要验证，并把实际结果写入收尾说明。

如果用户要求“持续消化 backlog”但没有指定下一个具体工作项，主 Agent 应优先把只读 explorer 用作候选发现器。每个 explorer 聚焦一个独立问题域，返回可复现现象、证据路径、建议验收检查、风险边界。主 Agent 只从这些候选中选择一个当前用户故事进入实现；未选候选必须写入来源 backlog、当前计划或下一轮候选说明。

## 4. Worker 分发提示

分发任何可写 worker 前，主 Agent 必须先记录 `git status --short`，并在提示中列出允许写入路径。worker 返回后，主 Agent 必须用 `git diff -- <允许路径>` 和 `git status --short` 核对实际写入范围；如果发现越界改动，先隔离并审查，不得直接合并到当前切片。

Worker 提示必须包含：

- 业务目标和验收条件。
- 允许读取和修改的文件范围。
- 禁止修改的文件范围。
- 需要遵循的 TDD / 验收流程和验证命令。
- 必须遵守 `AGENTS.md`、TDD、禁止回滚他人改动、禁止 `as any` / `@ts-ignore` / 静默 fallback。
- “你不是唯一改代码的 agent；不要回滚或覆盖他人改动；如发现范围外问题，只报告不要修”的约束。
- 完成后必须列出读取文件、修改文件、验证命令与结果、未运行验证、建议提交边界、残余风险和未完成项。
- 统一状态：`DONE`、`DONE_WITH_CONCERNS`、`NEEDS_CONTEXT` 或 `BLOCKED`。

## 5. 当前工作区并行协作

目标模式默认不使用 worktree。并行协作发生在当前主工作区的控制边界内：只读 explorer / verification 可以积极并行；可写 worker 必须有严格文件所有权和验收步骤。

推荐调度方式：

1. 主 Agent 作为 dispatcher / integrator，负责拆任务、维护共享 todo / playbook / plan、审查结果、运行最终验证和提交。
2. 只读 explorer / verification 不得修改文件；它们返回证据、风险和建议。
3. 可写 worker 只能修改分发提示列出的文件集合，不得修改共享记录文件、playbook、全局配置或高冲突文件，除非主 Agent 明确把这些文件分配给它。
4. 多个 worker 可以并行启动，但写入范围必须互不重叠。高冲突文件，例如 `ArtifactPane.tsx`、大型 store、共享 todo 和 playbook，默认由主 Agent 最后统一接线。
5. worker 返回后，主 Agent 按顺序集成：先读最终说明，再检查实际 diff 和写入范围，再决定是否保留、调整或拒绝该结果。
6. 每接受一个 worker 结果，运行该切片最小验证；全部接受后运行相关测试、lint、build 和 `git diff --check`。

并行子智能体只能降低探索和局部实现时间，不能消除最终语义冲突。若两个 worker 触达同一设计面或同一用户流程，主 Agent 必须在集成阶段重新审查行为一致性，不得仅凭 diff 无冲突判定完成。

## 6. 验收子智能体结果

子智能体验收至少包含四步：

1. 先读其最终说明，确认是否有 `NEEDS_CONTEXT`、`BLOCKED`、未运行验证或残余风险。
2. 对照分发提示核对实际改动是否越界，读取相关 diff，而不是只信任子智能体摘要。
3. 由主 Agent 在当前上下文重新运行本切片最小验证；如环境无法运行，必须记录阻塞原因和替代验证。
4. 将未采纳候选和残余风险写入来源 backlog、当前计划或收尾说明，确保并行探索结果不会只停留在对话上下文中。

只有这些验收步骤都完成后，子智能体结果才能进入完成记录。子智能体输出必须被当作工程输入，而不是完成证明。任何“已修复”“测试通过”“可合并”的说法，都必须由主 Agent 在当前上下文重新核对证据后才能对用户声明。

## 7. 不要使用子智能体的场景

- 问题尚未分清边界，多个症状可能来自同一个根因。
- 任务需要连续修改同一文件或共享状态，多个 worker 会互相覆盖。
- 多个子智能体会修改同一文件、同一 store 状态机、同一 API schema 或同一测试文件。
- 还没有完成根因定位，只是对同一症状提出多个猜测性修复。
- 只是单行修复、单个聚焦测试或主 Agent 已经持有完整上下文，分发成本高于收益。
- 子智能体需要凭证、联网、浏览器登录态或外部权限，而主 Agent 尚未取得用户授权。
- 当前执行环境没有可用子智能体或任务分发能力；此时不要强行模拟并行，改为顺序执行只读探索或实现，并记录降级原因。
