# 目标模式 CGA 与切片门禁附录

本文承载目标模式的 Current State Gap Analysis 模板、切片厚度门禁、过薄切片处理方式和进度口径。主执行入口见 `docs/strategy/goal-mode-playbook.md`。

## 1. CGA 目标

CGA 的目标是基于最新代码、测试、文档和用户目标，判断下一步最有价值、最可验收、最少破坏当前契约的工作包。

CGA 不是把 backlog 拆得更细的任务分拣器，而是把分散缺口重新组装成可交付能力包的决策过程。默认先问“用户要完成哪段真实任务”，再决定需要一起纳入哪些后端契约、前端入口、状态承接、错误恢复、下游消费和验证证据。

每轮 CGA 必须包含：

- 事实源快照：本轮实际读取了哪些关键文件。
- 能力包聚合：把 todo 子项、用户反馈、测试失败和技术缺口归并到同一用户场景或工程信任闭环下。
- 候选 gap：至少 2 个候选能力包；如果确实只有 1 个，也要说明为什么其他缺口无法在本轮形成能力包。
- 排序依据：为什么选中的 gap 比其他候选更适合作为本轮用户故事。
- 用户功能包边界：明确本轮代表的完整用户功能，包含哪些同一用户意图下的相邻缺口，排除哪些缺口及理由。
- 切片厚度门禁：明确是否通过“入口、动作、处理、结果、承接、失败反馈、证据”检查。
- 未选候选去向：保留为下一轮候选、写入来源 backlog / 专项计划、归入更大工作包、标记 blocked / wont-do，或明确暂不处理。
- 验收口径：每个候选都要说明可以被什么证据验证。

### 已确认目标轮次的承接检查

如果用户已经确认了目标轮次、阶段顺序和验收边界，并且这些内容已经写入 `docs/todos/`、专项计划或上一轮 spec / plan，后续执行该序列时不需要重复完整候选排序型 CGA。此时必须做轻量目标承接检查，确认“继续执行已声明的下一轮”仍然成立。

承接检查必须包含：

- 事实源快照：本轮重新读取了哪些关键文件、todo、spec / plan、代码和测试。
- 已确认目标来源：引用哪份 todo、计划或用户确认记录，以及本轮对应第几轮 / 哪个阶段。
- 改道条件检查：是否出现新的 P0/P1、未关闭质量门、LLM judge 低分、测试失败、架构冲突、用户新目标 / 纠错反馈、工作区冲突或外部阻塞。
- 质量门记录：若本轮或上一轮引用了 judge、E2E、全量自动化或人工审查结果，记录分数 / 状态、通过线、失败维度、修复或暂缓决定。
- 边界复核：本轮仍覆盖哪个用户故事，排除哪些后续轮次，是否需要拆分或合并。
- 验收复核：本轮仍用哪些测试、命令、artifact、日志或截图证明。
- 结论：继续承接 / 升级为完整 CGA / 暂缓并说明原因。

一旦承接检查发现目标、顺序、代码事实或风险排序发生实质变化，必须停止轻量流程，恢复完整 CGA。

## 3. 切片厚度门禁

目标模式的默认切片单位是“完整用户功能厚切片”，不是“代码改动集合”。完整用户功能厚切片必须围绕一个真实用户目标命名，并包含必要的入口、操作、系统处理、可见结果、状态承接、失败恢复和证据。

默认通过标准：

- 有明确用户入口或调用方入口。
- 有清晰的用户目标和完成状态，而不是只说明要改哪个模块。
- 有一个用户或调用方会主动触发的真实动作，而不是单纯内部预备工作。
- 系统完成可描述的核心处理，并产生可观察结果。
- 结果能被保存、恢复、下游消费、评判或作为后续证据承接。
- 失败路径能显式停止并说明原因，不靠 silent fallback。
- 验证证据覆盖入口、处理、结果和关键失败路径。
- 完成后能用一句“用户现在可以...”或“调用方现在可以...”说清楚新增能力。

默认不通过的信号：

- 用户故事名称只能写成“新增字段”“补一个 endpoint”“修一个 parser”“调整一个按钮”“补一条测试”。
- 用户故事名称只能写成某个内部算法分支或局部异常场景，例如“某类段落 merge”“某种 Mermaid 语法变体”“某个 provider 错误码”。
- 验收条件只验证内部函数，不验证 API、页面、工作流、下游消费或可见反馈。
- 完成后只能汇报百分比增加 0.x%，但说不清用户多完成了什么。
- 当前切片显然只是某个用户动作链的前半段，而后半段没有技术阻断却被留到下一轮。
- 需要连续多轮目标模式才能让用户第一次看到或使用同一能力。

不通过时的处理顺序：

1. 先扩大边界，把同一用户场景下相邻的后端契约、前端服务、页面交互、状态持久化、错误反馈、下游消费和验证证据并入同一用户故事。
2. 再检查扩大后的用户故事是否仍是同一个用户目标。只要入口、动作、状态承接和失败恢复仍服务于同一目标，默认保持合并。
3. 如果扩大后会跨越不相关用户场景或显著增加破坏面，把该候选记录为更大能力包的子项，并改选更合适的用户故事。
4. 如果它是工程信任闭环例外，必须写清楚解除的阻断、调用方受益、后续能力如何依赖它，以及为什么不能与用户可见能力同轮交付。
5. 如果仍无法形成真实能力包，暂缓，不进入 spec / plan。

## 4. 优先级口径

优先级默认如下：

1. 用户最新明确目标。
2. 阻断主路径演示、使用或部署的 P0 问题。
3. 会导致 silent fallback、协议混用、密钥泄露、错误写入或证据失真的工程风险。
4. `docs/todos/` 中 P0 且能提升系统专业可信度、能力评判或后续目标模式选题质量的工作项。
5. 当前活跃专项计划或 backlog 中高优先级且可形成纵向切片的工作项。
6. 能提高后续目标模式迭代可信度的测试、验证和文档护栏。
7. 普通清理、性能优化和历史文档整理。

一个合格用户故事应满足至少一项：

- 交付一个用户可直接使用的新功能或完整体验改进。
- 打通一条原本断裂的主路径或跨模块链路。
- 把一个只读能力升级为可操作、可保存、可恢复或可评判的闭环。
- 把一组分散的底层能力集成为调用方可复用的 API / 页面 / 工作流能力。
- 消除会造成 silent fallback、错误写入、安全风险、证据失真或无法验证的工程阻断。

## 5. 常见过薄切片

| 过薄切片 | 为什么过薄 | 应并入的厚切片 |
| --- | --- | --- |
| 只新增 schema / 类型 | 只有机器契约，没有用户可见结果和消费链路 | API / SSE / 前端状态写入 / 测试证据闭环 |
| 只修一个 parser 分支 | 用户只获得错误分类，主流程没有新增可理解能力 | 请求边界、错误文案、端点测试和 UI 可见错误 |
| 只改 prompt | LLM 输入改变但用户价值不可证明 | artifact 质量、契约校验、真实或 mock 输出验收 |
| 只补测试 | 验证加强但能力不变 | 对应用户故事的 TDD / 验收任务；工程信任闭环例外需说明独立价值 |
| 只更新文档措辞 | 不改变后续执行行为或当前能力边界 | 当前主路径文案、playbook 规则或稳定事实源校准 |
| 只新增一个按钮或入口 | 用户只能到达半成品页面或触发单点动作，没有完整处理链 | 页面加载、状态展示、核心操作、错误处理和返回路径 |
| 只补一个 Artifact 三方 merge 场景 | 用户真正需要的是编辑与生成并发时的可靠承接，不是某个算法分支 | Artifact 冲突处理能力包：安全自动合并、危险场景拦截、可读原因、手工承接、保存 / 恢复和验证证据 |
| 只补一个 Mermaid 语法兼容分支 | 用户真正需要的是生产环境可视化稳定显示与失败可恢复，不是单条语法规则 | 可视化稳定呈现能力包：契约约束、安全渲染、超时、fallback、重试和生产复现证据 |
| 只补一个模型供应商错误码 | 用户真正需要的是知道为什么失败并能完成配置修复，不是看到更多内部分类 | 模型配置自检与修复闭环：异常归因、设置入口、连接检测、保存 / 轮换、重试和运行统计 |

受控例外：如果某个底层缺口会导致后续工作出现 silent fallback、安全风险、证据失真或无法验证，允许作为工程信任闭环进入用户故事。但它仍必须给调用方或后续目标模式提供独立价值，不能停在内部 helper。

## 6. Superpowers 自问自答记录

能力包级目标模式必须真正执行 Superpowers `brainstorming` 的需求细化过程，而不是把结论包装成几条问答。执行时把 skill 原本要求向用户逐步询问的问题，改成 Agent 基于当前仓库事实、todo、代码、测试和用户目标的自问自答，并在 spec 中保留可审计记录。

最低记录结构：

- **Explore Project Context**：当前代码、文档、测试、git 状态和相关历史分支说明了什么；当前需求是否过大需要拆分。
- **Visual Companion Decision**：本轮是否涉及视觉设计问题；如果不需要，说明原因。
- **Clarifying Questions**：围绕用户是谁、要完成什么、成功状态、输入来源、约束、失败路径、下游承接和不做事项逐项回答。
- **Approaches**：列出 2-3 个实现 / 产品方案、各自 trade-off、推荐方案和不选理由。
- **Presented Design**：形成 architecture、components、data flow、error handling、testing 的设计结论。

如果自问自答只是在 spec 中补一句“已思考用户意图和边界”，或只列少量结论而没有按上述问题链展开，视为未完成 Superpowers brainstorming，不能继续写 implementation plan 或进入 TDD。

## 7. 进度口径

目标模式汇报进度时，不按文件数、测试数、提交数或拆分出的微任务数估算，也不要用过细切片制造 0.x% 的进度增量。进度应基于剩余用户可感知能力包、主路径闭环和工程信任闭环的完成情况估算。

汇报百分比时必须同时说明：

- 已完成的用户可感知能力包有哪些。
- 剩余的能力包还有哪些，不把子字段、子按钮、单端点当作独立百分比单位。
- 当前选择的下一切片预计会交付哪一个完整能力包。
- 如果百分比超过 90%，仍要列出剩余能力包，避免给出“做完下一片就全部完成”的误导。

如果用户和 CGA 已经确认“剩余 N 个能力包”，后续恢复目标模式时应以这 N 个能力包作为候选粒度，而不是重新把它们拆回 todo 子项。CGA 可以重新排序、合并或暂缓能力包，但必须说明原因。
