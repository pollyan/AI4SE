# 需求模式知识库

## 常见需求模式识别

### 用户管理模式

**典型特征**：
- 用户注册、登录、个人资料管理
- 角色权限控制和访问管理
- 用户行为跟踪和偏好设置

**常见需求**：
- 多种登录方式（邮箱、手机、第三方）
- 忘记密码和账户恢复
- 用户权限分级管理
- 个人信息安全和隐私保护

**澄清重点**：
- 用户类型和角色划分
- 权限控制粒度
- 安全合规要求
- 第三方集成需求

### 内容管理模式

**典型特征**：
- 内容的创建、编辑、发布、管理
- 版本控制和审批流程
- 内容分类和搜索

**常见需求**：
- 富文本编辑器功能
- 多媒体文件上传和管理
- 内容审核和发布流程
- SEO优化和元数据管理

**澄清重点**：
- 内容类型和结构
- 审批流程和权限
- 发布渠道和时机
- 内容生命周期管理

### 工作流程管理模式

**典型特征**：
- 任务创建、分配、跟踪
- 流程定义和状态管理
- 协作和沟通功能

**常见需求**：
- 灵活的工作流定义
- 任务优先级和截止日期
- 团队协作和通知
- 进度报告和统计

**澄清重点**：
- 业务流程复杂度
- 角色职责划分
- 自动化需求程度
- 集成其他系统需求

### 数据分析和报告模式

**典型特征**：
- 数据收集、处理、分析
- 报告生成和可视化
- 趋势分析和预测

**常见需求**：
- 实时数据监控
- 自定义报告模板
- 数据导出和分享
- 预警和自动通知

**澄清重点**：
- 数据来源和质量
- 分析维度和指标
- 报告受众和用途
- 数据更新频率

## 需求质量评估标准

### SMART原则应用

**Specific（具体性）**
- 需求描述是否明确具体？
- 避免模糊的形容词（如"更好"、"快速"、"用户友好"）
- 包含具体的操作和结果

**Measurable（可测量性）**
- 是否定义了明确的成功标准？
- 能否通过测试验证完成？
- 包含量化的性能指标

**Achievable（可实现性）**
- 在给定资源和时间内是否可实现？
- 技术可行性是否确认？
- 团队能力是否匹配？

**Relevant（相关性）**
- 是否与业务目标相关？
- 是否解决真实用户问题？
- 优先级是否合理？

**Time-bound（有时限性）**
- 是否有明确的时间要求？
- 交付节点是否现实？
- 依赖关系是否考虑？

### 需求完整性检查清单

**功能完整性**
- [ ] 主要用例是否都已识别？
- [ ] 异常流程是否都已考虑？
- [ ] 边界条件是否已定义？
- [ ] 业务规则是否完整？

**非功能完整性**
- [ ] 性能要求是否明确？
- [ ] 安全要求是否充分？
- [ ] 可用性标准是否定义？
- [ ] 兼容性需求是否考虑？

**接口完整性**
- [ ] 用户界面需求是否清晰？
- [ ] 系统接口是否定义？
- [ ] 数据接口是否明确？
- [ ] 第三方集成是否考虑？

### 需求一致性验证

**内部一致性**
- 需求之间是否存在冲突？
- 优先级设置是否合理？
- 约束条件是否现实？

**外部一致性**
- 与业务目标是否一致？
- 与用户期望是否匹配？
- 与技术架构是否兼容？

## 行业最佳实践

### 敏捷需求管理

**用户故事最佳实践**
- 使用标准格式："作为...我希望...以便..."
- 保持故事独立性和可测试性
- 控制故事规模，通常1-3天完成
- 定义清晰的验收标准

**Epic分解策略**
- 按用户价值流分组功能
- 考虑技术依赖和实现边界
- 保持Epic在2-8个故事规模
- 优先安排能快速验证假设的Epic

### 需求变更管理

**变更控制流程**
1. 变更请求提交和记录
2. 影响分析和评估
3. 变更决策和批准
4. 变更实施和验证
5. 变更结果评估

**变更影响评估**
- 功能影响范围分析
- 技术实现复杂度评估
- 时间和成本影响计算
- 风险和依赖关系分析

### 需求可追溯性

**向前追溯**
- 需求 → 设计 → 实现 → 测试
- 确保每个需求都有对应实现
- 验证实现是否满足需求

**向后追溯**
- 测试 → 实现 → 设计 → 需求
- 确保每个功能都有需求依据
- 识别过度实现或偏离需求

### 干系人管理

**干系人识别**
- 主要用户群体
- 业务决策者
- 技术实施团队
- 运维支持团队
- 外部合作伙伴

**需求沟通策略**
- 针对不同角色使用不同沟通方式
- 定期需求评审和确认
- 及时反馈和问题解决
- 文档化所有重要决策

## 常见需求陷阱和避免策略

### 需求陷阱识别

**范围蔓延**
- 特征：需求不断增加，项目范围失控
- 原因：初期需求不清晰，变更控制不严格
- 避免：明确需求边界，建立变更控制流程

**金镀陷阱**
- 特征：追求完美功能，过度设计
- 原因：没有明确的MVP定义
- 避免：聚焦核心价值，分阶段交付

**技术驱动需求**
- 特征：从技术角度定义需求，忽视用户价值
- 原因：技术团队主导需求定义
- 避免：始终从用户价值出发定义需求

### 质量保证策略

**需求评审机制**
- 定期组织跨职能团队评审
- 使用检查清单确保完整性
- 邀请最终用户参与验证
- 记录和跟踪评审结果

**原型验证**
- 使用低保真原型快速验证概念
- 高保真原型验证用户体验
- 技术原型验证技术可行性
- 收集用户反馈并迭代改进

**持续验证**
- 在开发过程中持续验证需求
- 及时调整不合理的需求
- 收集用户使用反馈
- 基于数据优化需求优先级
