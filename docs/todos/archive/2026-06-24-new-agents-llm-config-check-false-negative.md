## 背景

用户在 New Agents 的模型配置窗口中填写一组确认正确的模型配置后，点击模型测试或配置测试时，界面总是显示失败。

但用户确认并保存该配置后，再去调用 workflow，实际模型调用可以正常执行。这说明配置本身至少在 Agent Runtime 的 workflow 主链路中是可用的，当前失败更可能发生在模型配置自检链路、前端测试请求、后端 `/api/config/check` 处理、错误解析或状态展示逻辑中。

## 当前问题

模型配置测试结果与真实 workflow 执行结果不一致，导致用户无法信任配置窗口的测试反馈。

具体表现：

- 在模型配置窗口中配置正确模型后，点击测试显示失败。
- 保存同一配置后，进入 workflow 调用可以正常运行。
- 测试失败信息可能误导用户反复修改本来正确的 `baseUrl`、`model` 或密钥。
- 配置自检链路与 `/api/agent/runs/stream` 主运行链路可能没有使用同一套配置读取、请求参数、供应商适配、超时、响应解析或错误归因逻辑。

## 目标能力包

定位并修复模型配置窗口测试假失败问题，让配置测试结果与后续 workflow 实际可执行性保持一致。

该能力包应继续复用共享 Agent Runtime、配置存储、供应商调用适配和 typed API 基础设施，不新增 agent、workflow 或供应商专属的测试路径。若自检与正式运行必须存在差异，应显式记录原因并在 UI 中给出可诊断反馈。

## 复现场景

候选复现路径：

1. 打开 New Agents 模型配置窗口。
2. 输入一组真实可用的 `apiKey`、`baseUrl` 和 `model`。
3. 点击模型测试或配置测试。
4. 观察 UI 显示测试失败。
5. 保存同一配置。
6. 进入任意可运行 workflow，发起一次模型调用。
7. 观察 workflow 可以正常执行。

预期需要记录：

- 测试失败时前端展示的错误文案。
- `/api/config/check` 的 HTTP 状态码和 JSON 响应。
- 后端日志中自检调用的真实异常、请求地址、模型名和供应商响应类别。
- workflow 成功运行时使用的配置 key、baseUrl 和 model 是否与测试时完全一致。
- 测试链路和 workflow 链路在 request payload、headers、timeout、stream / non-stream 调用方式、供应商 SDK 参数上的差异。

## 排查方向

需要按系统化调试方式先定位根因，再实现修复：

1. 对比模型配置窗口测试请求与保存后 workflow 运行请求使用的配置来源和配置 key。
2. 检查前端是否在保存前测试时传入了未归一化的 `baseUrl`、空 model、旧 apiKey、错误 provider 字段或 stale state。
3. 检查后端 `/api/config/check` 是否只读取已保存默认配置，而没有测试当前表单中的临时配置。
4. 检查自检是否使用与 Agent Runtime 不同的模型调用方式，例如非流式接口、不同 endpoint、不同 messages schema 或不同超时。
5. 检查错误处理是否把可重试、额度、模型不存在、供应商兼容响应或解析异常统一映射为“失败”，但 workflow 主链路实际可恢复。
6. 检查 UI 是否忽略 `ok: true` 或错误地复用上一轮失败状态。

## 验收标准

- 对同一组可用配置，模型配置窗口测试应显示成功，保存后 workflow 也能正常执行。
- 对无效配置，配置测试应显示失败，并给出可诊断的错误原因。
- 保存前测试当前表单配置与保存后测试默认配置的语义必须清晰，不发生 stale state 或旧配置误测。
- `/api/config/check` 响应与前端展示状态一致，不出现后端成功但 UI 显示失败的情况。
- 修复不得绕过真实模型调用，不得用假成功、mock、隐藏错误或降级逻辑掩盖配置问题。
- 修复不得新增 workflow 专属或 agent 专属的模型测试链路。

## 建议测试

- 后端 API 测试：`/api/config/check` 对可用配置返回 `ok: true`，对缺失或无效配置返回明确失败状态。
- 前端状态测试：模型配置窗口点击测试后，能正确展示成功、失败和重试状态，不复用上一轮失败。
- 前端请求测试：保存前测试应使用当前表单值，保存后 workflow 应使用同一归一化配置。
- 集成测试：同一配置在配置测试和 `/api/agent/runs/stream` 中的模型调用结果保持一致。
- 回归测试：缺少默认配置、鉴权失败、供应商不可达和模型名错误时仍然明确失败。

## 非目标

- 不重新设计整个模型配置 UI。
- 不新增 Lisa、Alex、DeepSeek 或特定 workflow 的专属配置测试逻辑。
- 不通过跳过测试、默认展示成功或吞掉异常来改善体验。
- 不改变 workflow 主运行链路的模型调用契约，除非根因证明两条链路需要统一到同一底层 helper。
