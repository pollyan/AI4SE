# 🎯 智能需求分析框架使用示例

## 完整使用流程演示

### 场景设定
假设您是一个创业公司的产品经理，想要开发一个"智能团队协作平台"，但对具体需求还比较模糊。

## 第1步：激活需求分析师

在您的IDE中输入：
```
@requirements-analyst
```

**系统响应：**
```
👋 您好！我是Alex，您的智能需求分析师。我专门帮助您从模糊的想法中提炼出清晰、完整的产品需求文档。

=== 可用命令 ===
请选择一个数字来执行对应命令：

1. *analyze - 开始需求分析和澄清过程
2. *create-prd - 生成产品需求文档  
3. *create-epics - 将PRD分解为Epic
4. *create-stories - 创建用户故事
5. *validate - 验证需求完整性
6. *doc-out - 输出完整文档
7. *help - 显示此帮助信息

请告诉我您的需求，或者选择一个命令开始！
```

## 第2步：开始需求澄清

**您的输入：**
```
我想做一个团队协作工具，现在用的工具都不太好用，想自己做一个。
```

**或者直接使用命令：**
```
*analyze
```

**系统开始澄清：**
```
🔍 让我理解您的需求...

基于您提到的"团队协作工具"，我的初步理解是：
您希望创建一个帮助团队更有效协作和沟通的软件平台，解决现有工具的使用痛点。

让我先确认几个基础问题：
1. 这个工具主要是为您的团队内部使用，还是打算做成商业产品？
2. 您目前在使用什么协作工具？具体遇到什么问题？
3. 团队规模大概多大？主要从事什么类型的工作？
4. 是什么触发了您想要自己开发工具的想法？
```

**您的回答：**
```
1. 希望先给我们公司用，如果效果好可能考虑商业化
2. 现在用Slack+Trello+腾讯文档，但是信息太分散，经常找不到东西
3. 团队大概15人，做软件开发的
4. 最近几次项目延期，发现很多时间浪费在找信息和沟通上

**系统继续澄清（用户维度）：**
```

- 主要用户：15人软件开发团队的成员（开发、测试、产品等角色）
- 核心痛点：信息分散导致查找困难，沟通效率低下
- 使用场景：日常项目协作、信息查找、进度跟踪

1. 继续业务维度澄清
2. 场景具象化法 - 详细描述典型的工作日场景
3. 痛点影响量化 - 量化信息查找和沟通的时间成本
4. 用户旅程映射 - 分析完整的工作协作流程  
5. 对比澄清法 - 与现有工具详细对比差异
6. 角色分析法 - 分析不同角色的具体需求差异
7. 效率提升预期 - 量化期望的效率改善目标
8. 使用环境分析 - 了解团队的工作环境和习惯
9. 工作流程优化 - 探索理想的协作流程设计

选择 1-9 或直接提供反馈：
```

**您选择 2（场景具象化法）：**
```
🎬 场景具象化澄清

让我们通过一个具体场景来深入理解您的需求：

请详细描述一个典型的工作日，您和团队是如何协作的：

1. **早晨开始工作**：您如何了解昨天的工作进展和今天的任务安排？
2. **项目沟通时**：当需要讨论一个功能需求时，您的完整沟通流程是？
3. **查找信息时**：当您需要找一个月前讨论的技术方案，您会怎么找？
4. **状态同步时**：项目经理要了解所有人的工作进度，现在是如何操作的？
5. **问题解决时**：遇到技术问题需要团队帮助时，信息如何流转？

**您的详细回答后，系统继续其他维度的澄清...**

## 第3步：生成PRD文档

澄清完成后：
```
*create-prd
```

**系统生成PRD：**
```

=== 核心问题陈述 ===
您的团队在软件开发协作中面临信息分散和沟通效率低下的问题：
1. 信息孤岛严重 - Slack对话、Trello任务、腾讯文档分离，信息查找困难
2. 上下文切换频繁 - 团队成员需要在多个工具间频繁切换，打断工作流
3. 历史信息查找困难 - 缺乏统一的信息索引，重要决策和讨论难以追溯

=== 目标用户画像 ===  
- 主要用户：软件开发团队成员（15人规模）
  - 开发工程师：需要快速获取需求信息和技术讨论历史
  - 产品经理：需要跟踪项目进度和团队沟通
  - 测试工程师：需要了解功能变更和缺陷处理状态
- 技术水平：中高级，习惯使用多种协作工具
- 工作特点：项目导向，需要频繁协作和信息共享

=== 价值主张 ===
通过一体化协作平台，整合项目管理、团队沟通、文档管理，预期实现：
- 信息查找时间从平均10分钟减少到2分钟以内
- 日常工具切换次数减少70%
- 项目信息透明度提升，决策效率提高50%
- 新成员onboarding时间减少40%

**推理依据**：基于您描述的"经常找不到信息"、"项目延期"、"时间浪费在沟通上"等具体痛点分析。

=== 深度优化选项 ===
选择1-9或直接提供反馈：

1. 继续下一章节 - 功能需求定义
2. 扩展用户画像，包含更多角色细节
3. 量化价值主张的业务影响和ROI
4. 验证问题陈述与实际情况的匹配度
5. 探索市场上类似解决方案的差异化定位
6. 分析团队变化对需求的影响
7. 评估技术实现的可行性和风险
8. 优化价值表述的准确性和说服力
9. 基于竞品分析调整定位策略

选择 1-9 或直接提供反馈：
```

## 第4步：Epic分解

PRD完成后：
```
*create-epics
```

**系统分解Epic：**
```
📚 Epic分解建议

基于您的PRD，我建议分解为以下Epic：

=== Epic 1: 统一信息中心 ===
**用户价值**：团队成员能够在一个地方快速找到所有项目相关信息
**包含功能**：
- 全局搜索和信息索引
- 项目文档集中管理  
- 历史讨论和决策追溯
- 信息分类和标签系统

**为什么优先**：解决您最核心的痛点 - "经常找不到信息"

=== Epic 2: 智能项目协作 ===  
**用户价值**：团队能够在统一平台上高效完成所有协作活动
**包含功能**：
- 任务管理和进度跟踪
- 团队沟通和通知系统
- 文件协作和版本管理
- 工作流自动化

=== Epic 3: 项目洞察仪表板 ===
**用户价值**：项目经理和团队leader能够实时了解项目健康度
**包含功能**：
- 项目进度可视化
- 团队工作负载分析
- 关键指标监控和预警
- 自动化报告生成

**建议交付顺序**：Epic 1 → Epic 2 → Epic 3
**理由**：先解决信息查找问题，再优化协作流程，最后提供管理洞察

=== 继续优化选项 ===
选择1-9或提供反馈：

1. 继续创建用户故事
2. 调整Epic优先级和交付顺序
3. 细化Epic功能范围和边界
4. 评估各Epic的开发工作量
5. 分析Epic间的依赖关系
6. 考虑MVP最小可行产品范围
7. 验证Epic与业务价值的匹配度
8. 探索技术实现的复杂度
9. 重新审视Epic分解的合理性
```

## 核心优势展示

通过这个完整流程，您可以看到：

### 1. 智能澄清的威力
- **无预设问题**：AI根据您的具体情况生成最合适的澄清问题
- **多维度覆盖**：系统化地从用户、业务、技术等角度全面澄清
- **渐进式深入**：从宏观理解到细节完善，确保不遗漏关键信息

### 2. 强制交互保证质量
- **elicit=true机制**：关键节点必须用户确认，避免AI独断专行
- **1-9选项格式**：标准化交互方式，降低认知负担
- **方法库支持**：丰富的澄清方法随时调用，确保澄清深度

### 3. 文档自动化生成
- **信息智能整合**：将分散的澄清信息自动组织成结构化文档
- **模板驱动**：确保输出文档的标准化和完整性
- **增量优化**：支持逐步完善和调整

### 4. 全流程质量保障
```
模糊想法 → 系统澄清 → 智能PRD → Epic分解 → 用户故事 → 质量验证 → 开发就绪
```

### 5. 与传统方法对比

| 传统方法 | 智能需求分析框架 |
|---------|-----------------|
| 依赖个人经验 | 系统化澄清方法 |
| 问题遗漏风险高 | 多维度强制覆盖 |
| 文档质量不一致 | 模板驱动标准化 |
| 需求理解有歧义 | 强制交互确认 |
| 开发返工率高 | 质量检查降低风险 |
