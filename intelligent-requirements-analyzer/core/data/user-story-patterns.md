# 用户故事模式库

## 标准故事模式

### CRUD操作模式

**创建模式**
```
作为 [用户角色]
我希望 能够创建[数据对象]
以便 [业务价值/目标实现]

验收标准：
- 用户可以输入所有必需字段
- 系统验证数据格式和完整性
- 创建成功后显示确认信息
- 数据保存到系统中
```

**读取/查看模式**
```
作为 [用户角色]
我希望 能够查看[数据对象]列表/详情
以便 [了解信息/做出决策]

验收标准：
- 数据按照合理的方式组织和显示
- 提供必要的筛选和搜索功能
- 分页处理大量数据
- 显示最新和准确的信息
```

**更新模式**
```
作为 [用户角色]
我希望 能够修改[数据对象]
以便 [保持信息最新/纠正错误]

验收标准：
- 可以修改允许编辑的字段
- 保留修改历史记录
- 验证修改权限
- 显示修改成功确认
```

**删除模式**
```
作为 [用户角色]
我希望 能够删除[数据对象]
以便 [清理无用信息/维护数据质量]

验收标准：
- 删除前显示确认提示
- 检查删除权限和约束
- 支持批量删除（如需要）
- 考虑软删除vs硬删除
```

### 工作流程模式

**任务分配模式**
```
作为 项目经理
我希望 能够将任务分配给团队成员
以便 确保工作的有序进行和责任明确

验收标准：
- 可以选择合适的团队成员
- 可以设置任务优先级和截止时间
- 被分配者收到通知
- 任务状态可以跟踪
```

**状态流转模式**
```
作为 [角色]
我希望 能够更新[工作项]的状态
以便 让团队了解工作进展

验收标准：
- 状态变更符合业务规则
- 状态变更有权限控制
- 相关人员收到状态变更通知
- 记录状态变更历史
```

**审批流程模式**
```
作为 审批者
我希望 能够审批[申请/请求]
以便 确保符合公司政策和标准

验收标准：
- 可以查看申请详细信息
- 可以批准或拒绝申请
- 可以添加审批意见
- 申请人收到审批结果通知
```

### 通知和沟通模式

**系统通知模式**
```
作为 [用户角色]
我希望 收到[相关事件]的通知

验收标准：

```

**消息发送模式**
```
作为 [发送方角色]
我希望 能够向[接收方]发送消息
以便 进行有效沟通和信息传递

验收标准：
- 消息可以准确发送给指定接收方
- 支持文本、文件等多种消息类型
- 消息有送达确认
- 消息有适当的隐私保护
```

## 故事质量检查模式

### INVEST原则检查模板

**Independent（独立性）检查**
- [ ] 故事可以独立开发和测试
- [ ] 不强依赖其他未完成的故事
- [ ] 可以独立部署和验证

**Negotiable（可协商性）检查**
- [ ] 实现细节可以在开发中讨论
- [ ] 留有设计和技术选择的空间
- [ ] 验收标准明确但实现灵活

**Valuable（有价值性）检查**
- [ ] 为最终用户提供明确价值
- [ ] 解决真实的用户问题
- [ ] 价值陈述清晰可理解

**Estimable（可估算性）检查**
- [ ] 开发工作量可以合理估算
- [ ] 技术复杂度明确
- [ ] 不确定因素已识别

**Small（小规模性）检查**
- [ ] 可在一个迭代内完成
- [ ] 通常1-3天开发时间
- [ ] 如果太大已适当拆分

**Testable（可测试性）检查**
- [ ] 有明确的验收标准
- [ ] 可以编写自动化测试
- [ ] 成功标准可验证

### 常见故事问题和修正

**问题：故事过大**
```
错误示例：
作为用户，我希望有一个完整的项目管理系统，以便管理所有工作。

修正方法：
拆分为具体的小故事：
- 作为项目经理，我希望创建新项目，以便组织团队工作
- 作为团队成员，我希望查看分配给我的任务，以便了解工作安排
- 作为项目经理，我希望跟踪项目进度，以便及时调整计划
```

**问题：故事缺乏价值陈述**
```
错误示例：
作为用户，我希望有一个登录页面。

修正方法：
作为注册用户，我希望能够安全登录系统，以便访问我的个人数据和执行相关操作。
```

**问题：故事过于技术化**
```
错误示例：
作为开发者，我希望实现REST API，以便前端调用。

修正方法：
作为移动应用用户，我希望能够在移动设备上查看我的账户信息，以便随时了解账户状态。
```
