# Epic分解任务

## 任务目的

基于完整的PRD需求文档，将功能需求智能分解为可管理的Epic，确保每个Epic都有明确的业务价值、合理的规模和清晰的验收标准。

## 前置条件

- 必须存在完整的PRD文档（requirements-prd.md）
- PRD中的功能需求已经过充分澄清和验证
- 用户画像、业务目标和技术约束已明确

## 分解执行步骤

### 第1步：PRD深度分析

**执行指令：**
1. **功能需求提取**
   - 从PRD中识别所有功能性需求
   - 分析每个功能的用户价值和业务价值
   - 理解功能间的逻辑关系和依赖

2. **用户价值流映射**
   - 根据用户旅程识别主要价值流
   - 将功能按照用户工作流进行逻辑分组
   - 确保每个价值流都能提供端到端的用户价值

3. **技术依赖分析**
   - 识别功能实现的技术依赖关系
   - 分析共享组件和基础设施需求
   - 考虑实现的自然边界和模块化

### 第2步：智能Epic分组

**分组原则：**

1. **用户价值导向**
   - 每个Epic必须对应一个完整的用户价值实现
   - Epic内的功能应该协同工作以实现特定用户目标
   - 避免技术导向的分组方式

2. **规模合理性**
   - 每个Epic的工作量应该在2-8个故事之间
   - Epic应该可以在1-3个迭代周期内完成
   - 避免过大或过小的Epic

3. **依赖最小化**
   - Epic之间的依赖关系应该尽可能少
   - 优先创建独立可交付的Epic
   - 必要的依赖应该明确标注

4. **MVP优先级**
   - 能够快速验证核心假设的Epic优先
   - 支撑最小可行产品的Epic应该识别出来
   - 增值功能和优化类Epic可以后置

### 第3步：Epic文档生成

**使用epic-tmpl.yaml模板生成每个Epic文档，包含：**

1. **Epic基础信息**
   - Epic名称和简短描述
   - 对应的用户价值主张
   - 业务目标和成功指标

2. **功能范围定义**
   - 包含的具体功能列表
   - 功能的详细描述和用户场景
   - 明确的范围边界（什么不包含）

3. **验收标准**
   - 可测试的验收条件
   - 用户满意度标准
   - 技术实现标准

4. **依赖和约束**
   - 对其他Epic的依赖关系
   - 技术约束和限制
   - 资源和时间约束

### 第4步：优先级排序和交付规划

**优先级评估维度：**

1. **用户价值影响**
   - 解决核心痛点的程度
   - 用户使用频率和重要性
   - 对用户体验的改善程度

2. **业务价值贡献**
   - 对业务目标的直接贡献
   - ROI预期和投资回报时间
   - 市场竞争优势

3. **技术可行性**
   - 实现复杂度和风险
   - 技术依赖的成熟度
   - 团队技能匹配度

4. **风险和不确定性**
   - 需求变更的可能性
   - 技术风险和挑战
   - 外部依赖的稳定性

**交付顺序建议：**
- 第一阶段：核心价值Epic（解决主要痛点）
- 第二阶段：增值Epic（提升用户体验）
- 第三阶段：优化Epic（性能和扩展性）

### 第5步：Epic质量验证

**使用epic-breakdown-checklist进行质量检查：**

- [ ] 每个Epic都有明确的用户价值主张
- [ ] Epic规模适中，可在合理时间内完成
- [ ] Epic之间依赖关系最小化
- [ ] 验收标准清晰可测试
- [ ] 优先级排序合理有依据
- [ ] 技术可行性已评估
- [ ] 风险和挑战已识别

## 输出交付物

1. **Epic清单文档**
   - 所有Epic的概览和优先级
   - Epic间依赖关系图
   - 建议的交付路线图

2. **详细Epic文档**
   - 每个Epic的完整规格说明
   - 存储在docs/epics/目录下
   - 文件命名：epic-{序号}-{名称}.md

3. **交付计划建议**
   - 分阶段交付策略
   - 里程碑和关键检查点
   - 风险缓解建议

## 与下游任务的衔接

Epic分解完成后，为用户故事生成任务准备：
- 每个Epic都有清晰的功能边界
- 用户场景和角色已明确
- 验收标准框架已建立
- 为story-generation任务提供充分输入
