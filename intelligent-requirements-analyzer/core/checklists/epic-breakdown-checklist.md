# Epic分解质量检查清单

## Epic定义质量

### 价值主张清晰性
- [ ] **Epic价值主张是否明确？**
  - 用户价值清晰可感知
  - 业务价值可量化衡量
  - 解决的问题具体明确
  - 预期成果可验证

- [ ] **价值实现是否可行？**
  - 实现路径清晰可行
  - 成功指标具体可测
  - 时间预期合理现实
  - 资源要求切实可达

### Epic范围合理性
- [ ] **Epic规模是否适当？**
  - 包含2-8个用户故事
  - 可在1-3个迭代内完成
  - 工作量估算在10-40故事点之间
  - 复杂度在团队能力范围内

- [ ] **Epic边界是否清晰？**
  - 包含功能明确列出
  - 排除功能明确说明
  - 与其他Epic边界清晰
  - 功能内聚性强

## 分解合理性检查

### 用户价值流对齐
- [ ] **Epic是否按用户价值流分组？**
  - 支撑完整的用户旅程
  - 功能协同提供用户价值
  - 避免技术导向的分组
  - 用户体验连贯一致

- [ ] **Epic是否解决完整问题？**
  - 覆盖问题的完整解决方案
  - 不存在功能缺失导致价值无法实现
  - 用户可以获得端到端的价值
  - 避免功能碎片化

### 技术实现合理性
- [ ] **Epic技术边界是否合理？**
  - 遵循系统架构的自然边界
  - 技术组件内聚性高
  - 跨系统依赖最小化
  - 技术债务影响可控

- [ ] **Epic实现复杂度是否均衡？**
  - 避免某个Epic过于复杂
  - 技术风险分布合理
  - 团队技能匹配度良好
  - 实现时间相对均衡

## Epic间关系检查

### 依赖关系管理
- [ ] **Epic间依赖是否最小化？**
  - 尽可能减少Epic间强依赖
  - 依赖关系明确标注
  - 依赖路径清晰可追踪
  - 循环依赖已消除

- [ ] **关键路径是否识别？**
  - 识别出影响整体进度的Epic
  - 关键路径的风险已评估
  - 关键路径有备选方案
  - 资源优先保障关键路径

### 交付顺序合理性
- [ ] **Epic优先级是否合理？**
  - 基于用户价值和业务价值排序
  - 考虑技术依赖关系
  - 风险和不确定性已考虑
  - MVP Epic明确标识

- [ ] **分阶段交付是否合理？**
  - 每个阶段都有独立价值
  - 阶段间逻辑关系清晰
  - 里程碑设置合理
  - 早期价值验证可行

## 检查结果评估

### 质量等级评定
- **优秀（90%+通过）**：Epic定义清晰，实施风险低，可以直接进入故事创建
- **良好（80-89%通过）**：大部分要求满足，少数问题需要快速修正
- **合格（70-79%通过）**：基本要求满足，需要制定改进计划
- **不合格（<70%通过）**：存在重大问题，需要重新分解或大幅修改

### 改进行动建议
- **立即修正**：影响Epic核心价值实现的问题
- **计划改进**：制定具体改进计划和时间表
- **持续优化**：纳入Epic实施过程中的持续改进
- **经验积累**：总结经验改进Epic分解方法

## 使用指导

### 检查时机
- Epic分解初版完成后
- Epic评审会议之前
- Epic实施计划制定前
- Epic范围变更之后

### 检查方法
- **自检**：Epic负责人使用清单自检
- **同行评审**：邀请其他产品经理或架构师评审
- **跨职能评审**：开发、测试、UX等团队共同评审
- **用户验证**：关键Epic邀请用户代表参与验证

### 检查工具
- 清单逐项检查记录表
- Epic质量评分卡
- 风险识别和评估表
- 改进行动计划模板
